# Bug 修复器

读取测试结果报告，按严重程度逐个定位并修复 Bug，修复后验证，输出修复报告。

## 触发词

修复 Bug、根据测试结果修复、修 bug、修bug

## 与上游的关联

本技能读取 `test-case-runner` 生成的测试结果报告，通过**文档ID**和**用例ID**实现全链路追溯。

| 关联字段 | 来源 | 用途 |
|---------|------|------|
| 源文档ID | test-results 文档头部 | 追溯到原始测试用例文档 |
| 用例ID | test-results 中每条失败用例 | 直接透传，不做二次编号，修复记录绑定原始用例ID |

## 执行流程

```
读取测试结果 → 按严重程度排序 → 逐个定位 → 修复 → 验证 → 输出修复报告
```

### 阶段 1：读取测试结果

1. 确定目标测试结果文档：
   - 用户指定了文档路径 → 直接使用
   - 用户指定了页面名称 → 查找 `test/test-results/{页面名}-test-results.md`
   - 用户未指定 → 扫描 `test/test-results/` 目录，列出所有文档供选择
   - 对话上下文中刚执行过 test-case-runner → 自动使用其输出的结果文档

2. 解析文档，提取：
   - 源文档ID（追溯原始测试用例）
   - Bug 汇总列表（用例ID、描述、严重程度、修复建议）
   - 失败用例详情（操作步骤、预期结果、实际结果、截图路径）

3. 按严重程度排序：Critical > High > Medium > Low

### 阶段 2：逐个定位

对每个 Bug：

1. 根据用例ID 定位相关代码：
   - 读取截图（如有），观察实际 UI 状态
   - 根据操作步骤和实际结果，定位问题代码文件
   - 读取关联的 Vue 组件、Store、API 文件
2. 分析根因：
   - 缺少错误处理 → catch 分支缺失
   - 缺少校验 → 前端/后端校验遗漏
   - 逻辑错误 → 条件判断/状态管理有误
   - UI 问题 → 样式/组件属性配置错误

### 阶段 3：修复

1. 按项目规则修复代码：
   - 遵循 `code-style.md`（禁止 any、禁止未使用声明）
   - 遵循 `vue-component.md`（TDesign 组件规范）
   - 遵循 `error-handling.md`（错误处理模式）
   - 遵循 `design-tokens.md`（样式令牌）
2. 修复原则：
   - **最小修改**：只修复 Bug 本身，不做额外重构
   - **不引入新问题**：修复不影响其他功能
   - **添加防护**：如果根因是缺少校验/错误处理，补上

### 阶段 4：验证

对每个修复：

1. 快速验证方式（按优先级选择）：
   - 运行相关单元测试（vitest）
   - 用 Playwright 重测该用例
   - 代码逻辑审查（无法自动验证时）
2. 验证结果：
   - 通过 → 标记"已修复 ✅"
   - 仍失败 → 标记"修复未生效 ❌"，附原因
   - 无法自动验证 → 标记"需手动验证 ⚠️"

### 阶段 5：输出修复报告

输出到 `test/fix-reports/{页面名}-fix-report.md`。

## 修复优先级策略

| 严重程度 | 修复策略 |
|---------|---------|
| Critical | 立即修复，修复后必须验证通过 |
| High | 优先修复，修复后必须验证 |
| Medium | 修复，验证方式可选 |
| Low | 修复，代码审查即可 |

## 修复原则

- **用例ID 透传**：修复记录中的用例ID 必须与 test-results 文档完全一致，不做二次编号
- **最小修改**：只修复 Bug 本身，不做额外重构或优化
- **遵循项目规则**：修复代码必须符合 code-style、vue-component、error-handling 等规则
- **不引入新问题**：修复前评估影响范围，确保不影响其他功能
- **可追溯**：每条修复记录都能通过用例ID → 源文档ID 追溯到原始测试用例
