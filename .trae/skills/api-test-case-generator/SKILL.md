# 接口测试用例生成器

扫描后端路由文件，生成结构化接口测试用例文档，覆盖 9 个常规维度 + 17 个特殊场景。

## 触发词

生成接口测试用例、API 测试用例、后端测试覆盖、接口测试文档

## 执行流程

```
确定目标模块 → 扫描路由/控制器/Service/数据层 → 生成测试用例 → 输出文档
```

### 阶段 1：确定目标模块

#### 用户指定了模块

直接使用，支持以下格式：
- 路由文件路径：`backend/src/routes/formulas.ts`
- 模块名称：`formulas`、`配方`
- 多个模块：逗号分隔

#### 用户未指定模块

采用**分层扫描**策略：

**索引层（自动，<1s）**：

1. 扫描 `backend/src/routes/*.ts`，快速统计每个文件的端点数
2. 按模块分组，输出选择清单：

```
扫描到 12 个路由模块：

【核心业务】
  1. formulas.ts      — 8 个端点（POST/GET/PUT/DELETE）
  2. materials.ts     — 12 个端点
  3. salesmen.ts      — 6 个端点
  4. sales.ts         — 6 个端点

【AI 系统】
  5. ai.ts            — 18 个端点
  6. agent.ts         — 15 个端点

...

建议：端点数 ≥8 的模块优先生成测试用例。
请输入编号（多选用逗号，all 全部，推荐 1,2,5）：
```

3. 用户选择模式与前端一致（单选/多选/all/关键词匹配）

4. 上下文推断：刚修改了某路由文件 → 推断为该模块

**深度层（选定后触发，2-5s/模块）**：

用户选定后，读取路由+控制器+Service+数据库表完整内容。

**批量模式优化**：与前端一致（并行扫描、增量缓存、优先级排序、进度提示）

### 阶段 2：扫描后端代码

| 层 | 文件 | 提取内容 |
|----|------|---------|
| 路由层 | `routes/*.ts` | HTTP 方法、路径、中间件（是否需认证） |
| 控制器层 | `controllers/*.ts` | 请求参数（query/body/params）、校验规则、响应结构 |
| Service 层 | `services/*.ts` | 业务逻辑、状态流转、计算公式、错误处理 |
| 数据层 | `config/database-better-sqlite3.ts` | 表结构、约束、外键、索引 |

对每个端点提取：

```
端点编号 | HTTP方法 | 路径 | 认证要求 | 请求参数 | 响应结构 | 关联表 | 业务逻辑
```

### 阶段 3：生成测试用例

#### 常规维度（9 个，每个端点都生成）

| 维度 | 前缀 | 说明 |
|------|------|------|
| **正向流程** | P | 标准请求，验证 200/201 + 响应结构 |
| **异常流程** | E | 参数错误、认证失败、服务异常、重复操作 |
| **边界条件** | B | 空值、超长、0、负数、特殊字符、SQL 注入 |
| **权限认证** | R | 未登录、不同角色、Token 过期、无权限 |
| **参数校验** | V | 必填缺失、类型错误、格式不合法、超出范围 |
| **状态流转** | S | 不同数据状态下操作的允许/禁止 |
| **数据一致性** | DC | 操作后数据库状态正确、关联数据一致 |
| **幂等性** | I | 重复请求结果一致（PUT/DELETE） |
| **数据隔离** | DI | 不同用户数据不可互访 |

每个用例包含：

| 字段 | 说明 |
|------|------|
| 用例ID | 端点编号 + 维度缩写 + 序号（如 A01-P01、A01-E01） |
| 用例名称 | 简明描述测试场景 |
| 前置条件 | 执行前必须满足的状态（含测试数据准备） |
| 请求参数 | 方法、路径、headers、query/body/params |
| 预期状态码 | 如 200、201、400、401、403、404、409、500 |
| 预期响应 | 响应体结构和关键字段值 |
| 预期数据库变化 | 操作后数据库应发生的变化（仅增删改用例） |

#### 特殊场景（17 个，按条件生成）

| 场景 | 前缀 | 触发条件 | 用例内容 |
|------|------|---------|---------|
| 分页排序 | X-PG | 分页参数（page/pageSize） | 翻页、越界、排序字段、总数一致 |
| 并发冲突 | X-CC | 更新/删除操作 | 同时修改同一记录、乐观锁 |
| 文件上传 | X-UP | upload 端点 | 格式校验、大小限制、无文件、损坏文件 |
| 批量操作 | X-BT | 批量端点（批量删除/导入） | 空数组、超大批量、部分失败 |
| 级联操作 | X-CS | 外键关联 | 删除被引用记录、级联更新/删除 |
| 计算准确性 | X-C | 计算逻辑（营养成分、利润率） | 计算结果与预期值对比 |
| 缓存行为 | X-CH | 缓存逻辑（parse_results） | 缓存命中、缓存过期、缓存与 DB 不一致 |
| 异步任务 | X-AS | 异步端点（导出/AI 解析） | 任务创建、进度查询、超时处理 |
| 请求方法限制 | X-MD | 路由定义 | 错误 HTTP 方法应返回 405 |
| 审计日志 | X-AL | audit_log / review_logs | 操作后审计日志是否正确记录 |
| 错误信息安全 | X-SE | 错误处理代码 | 500 错误不泄露堆栈/SQL |
| 响应格式一致性 | X-RF | 多个端点 | 成功/失败响应格式统一 |
| 限流/频率控制 | X-RL | rate limit / daily_call_limit | 超频请求返回 429 |
| Content-Type 校验 | X-CT | POST/PUT 端点 | 发送错误格式请求体被拒绝 |
| 请求体大小限制 | X-LB | body parser 配置 | 超大请求体被拦截 |
| SQLite 并发写入 | X-SQ | 写操作（INSERT/UPDATE/DELETE） | 多个写请求同时到达不丢数据 |
| AI 供应商故障转移 | X-AF | agent_provider_health / fallback_configs | 主供应商不可用时自动切换 |

## 注意事项

- **文档ID 透传**：文档ID 是下游 api-test-runner 和 bug-fixer 的唯一追溯标识
- **文件hash 校验**：下游技能启动时通过对比 hash 判断用例文档是否变更
- **用例ID 透传**：结果文档和修复报告中的用例ID 必须与本文档完全一致
- **测试数据独立**：每条用例的前置条件应包含完整的数据准备步骤，不依赖其他用例的副作用
- **遵循 api-design 规则**：测试用例的预期响应格式应与 `.trae/rules/api-design.md` 一致
- **遵循 error-handling 规则**：错误响应的验证应与 `.trae/rules/error-handling.md` 一致
