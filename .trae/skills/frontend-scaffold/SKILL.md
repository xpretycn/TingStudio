# 前端页面开发骨架规范

本 Skill 确保 AI 在编写前端业务代码前，先完成骨架搭建，避免风格混乱、代码冗余、组件重复等问题。

## 触发条件

- 用户要求新建前端页面或模块
- 用户要求搭建前端新功能
- 用户说"新建页面"、"添加模块"、"创建视图"等

## 执行流程

**严格按以下 4 步顺序执行，不得跳过。**

### Step 1: 确认设计风格

1. 扫描项目中已有的设计 Token / 主题变量文件（如 `tokens.scss`、`theme.ts`、CSS 变量等）
2. 若项目已有设计体系，直接引用并输出当前风格摘要
3. 若无设计体系，引导用户确定：
   - 风格定位（简洁 / 商务 / 科技感 / 工具化）
   - 风格参考（可借鉴的设计体系或网站截图）
   - 将风格具体化为：颜色体系、字体、间距、圆角等设计元素
4. **输出**：设计风格确认清单

### Step 2: 确认技术方案与组件库

1. 读取 `package.json` 确认前端框架和 UI 组件库
2. 列出组件库中可用的组件清单，明确：
   - 哪些 UI 元素直接使用组件库组件
   - 哪些需要基于组件库封装业务组件
   - 哪些需要完全自定义实现
3. 确认路由方案、状态管理方案、API 层方案
4. **输出**：技术栈确认 + 组件使用边界

### Step 3: 规划目录、模块与组件

1. 按项目约定目录结构放置文件（读取 `.trae/rules/` 下相关规则）
2. 划分模块边界，避免代码交叉混杂
3. 遵循 DRY 原则：**同类 UI 结构出现超过 2 次，必须抽象为可复用组件**
4. 优先使用选定的组件库，不足时再封装特定业务组件
5. **输出**：
   - 文件清单（页面、组件、API、Store、类型定义等）
   - 组件复用计划（哪些需抽公共组件）

### Step 4: 准备样式系统

1. 检查设计 Token 是否就位（CSS 变量 / SCSS 变量 / 主题配置）
2. **严禁硬编码样式数值**，所有颜色、字号、间距、圆角必须使用 Token 变量
3. 若有国际化需求，文案统一管理，避免硬编码文本
4. 主题色、深色/浅色模式等关键样式变量集中管理
5. **输出**：样式变量引用方式说明

## 验收 Checklist

骨架搭建完成后，必须逐项确认：

- [ ] 项目能正常启动运行
- [ ] 路由配置正确，新页面可访问
- [ ] 目录结构符合项目约定
- [ ] UI 组件库正确集成并可用
- [ ] 设计 Token 已应用，无硬编码样式值
- [ ] 基础布局组件（Header、Sidebar 等）已搭建
- [ ] 公共组件已按 DRY 原则抽象

**未通过全部 Checklist 项，不得进入业务代码编写阶段。**

## 禁止行为

- **禁止跳过骨架流程直接写业务代码**
- **禁止硬编码颜色、字号、间距等样式值**，必须使用 Token 变量
- **禁止重复实现组件库已提供的功能**
- **禁止在无设计 Token 的情况下开始编码**
- **禁止同类 UI 结构重复实现超过 2 次而不抽象为公共组件**

## 与项目规则联动

本 Skill 执行时应同步读取以下项目规则（若存在）：

- `.trae/rules/code-style.md` — 代码风格规范
- `.trae/rules/vue-component.md` — Vue 组件规范
- `.trae/rules/eslint-check.md` — ESLint 检查规范
- `.trae/rules/project-ui-dynamic-theme.md` — 主题变量规范

确保骨架产出与项目既有规范保持一致。
