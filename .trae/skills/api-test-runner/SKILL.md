# 接口测试执行器

读取接口测试用例文档，用 HTTP 请求逐条执行，验证响应、数据库和性能，输出测试结果报告。

## 触发词

跑接口测试、执行 API 测试、后端测试、跑接口

## 与上游的关联

| 关联字段 | 来源 | 用途 |
|---------|------|------|
| 文档ID | api-test-cases 文档头部 | 唯一标识，写入结果文档头部 |
| 用例ID | api-test-cases 中每条用例 | 直接透传，不做二次编号 |
| 文件hash | api-test-cases 文档头部 | 校验用例文档是否变更 |

## 执行流程

```
读取测试用例 → 变更校验 → 启动后端服务 → 准备测试数据 → 逐条执行 → 记录结果 → 输出报告
```

### 阶段 1：读取测试用例

1. 确定目标文档：
   - 用户指定了文档路径 → 直接使用
   - 用户指定了模块名称 → 查找 `test/api-test-cases/{模块名}-api-test-cases.md`
   - 用户未指定 → 扫描 `test/api-test-cases/` 目录列出选择
   - 对话上下文中刚执行过 api-test-case-generator → 自动使用其输出

2. 解析文档，提取文档ID、文件hash、所有端点及其用例

### 阶段 2：变更校验

与前端一致：计算 hash 对比，不一致时警告；已有结果文档时增量执行。

### 阶段 3：启动后端服务

1. 检查后端服务是否运行（默认 `http://localhost:3000`）
2. 未运行 → 启动 `npm run dev`（后端）
3. 等待服务就绪（健康检查 `/health` 返回 200）
4. 准备认证 Token：
   - 使用测试账号登录获取 Token
   - 或直接生成 JWT Token（如有权限）

### 阶段 4：准备测试数据

1. 清理残留测试数据（DELETE WHERE name LIKE '[test]%')
2. Seed 基础数据（测试用户、测试原料、测试配方等）
3. 每条用例的前置条件中要求的数据，通过 API 或直接操作 DB 准备

### 阶段 5：逐条执行

对每条用例：

1. **准备前置条件**：确保数据状态满足要求
2. **发送请求**：按用例定义的方法、路径、headers、body 发送 HTTP 请求
3. **验证响应**：
   - 状态码是否与预期一致
   - 响应体结构是否正确（字段存在、类型正确）
   - 响应体关键字段值是否与预期一致
4. **验证数据库**（仅增删改用例）：
   - 查询数据库，验证数据变化是否符合预期
   - 验证关联表数据（外键、级联）
5. **记录结果**：
   - 通过 ✅ → 记录"通过" + 响应时间
   - 失败 ❌ → 记录实际状态码、实际响应、与预期的差异
   - 跳过 ⏭ → 记录跳过原因

#### 多层验证策略

| 验证层 | 验证方式 | 适用用例 |
|--------|---------|---------|
| **响应层** | 状态码 + 响应体结构 + 字段类型 + 关键字段值 | 所有用例 |
| **数据层** | 操作后查询数据库，验证数据正确性 | 增删改用例（P/E/B/DC 类） |
| **关联层** | 验证关联表数据（外键、级联、审计日志） | X-CS/X-AL 类用例 |
| **性能层** | 响应时间 < 阈值 | 所有用例 |

#### 性能阈值

| 端点类型 | 默认阈值 |
|---------|---------|
| 普通 CRUD | 500ms |
| 列表查询（含分页） | 1000ms |
| AI 相关接口 | 30000ms |
| 文件上传 | 5000ms |
| 导出 | 10000ms |

#### 特殊场景执行策略

| 场景 | 执行方式 |
|------|---------|
| **权限认证 (R)** | 切换不同角色 Token，验证 401/403 |
| **幂等性 (I)** | 同一请求发送两次，验证第二次结果 |
| **数据隔离 (DI)** | 用户 A Token 请求用户 B 的数据，验证不可见 |
| **分页排序 (X-PG)** | 请求多页数据，验证无交集、总数一致 |
| **并发冲突 (X-CC)** | 两个并发请求同时发送（Promise.all） |
| **文件上传 (X-UP)** | 发送 multipart/form-data，含各种文件 |
| **批量操作 (X-BT)** | 发送包含大量 ID 的数组 |
| **级联操作 (X-CS)** | 删除后查询关联表验证级联 |
| **计算准确性 (X-C)** | 提交已知数据，验证响应中的计算结果 |
| **缓存行为 (X-CH)** | 重复请求验证缓存命中（第二次更快或响应标记） |
| **异步任务 (X-AS)** | 创建任务后轮询进度接口 |
| **请求方法限制 (X-MD)** | 用错误 HTTP 方法请求，验证 405 |
| **审计日志 (X-AL)** | 操作后查询 audit_log 表验证记录 |
| **错误信息安全 (X-SE)** | 触发 500 错误，验证响应不含堆栈/SQL |
| **响应格式一致性 (X-RF)** | 对比多个端点的响应格式 |
| **限流/频率控制 (X-RL)** | 短时间发送超限请求，验证 429 |
| **Content-Type 校验 (X-CT)** | 发送 XML 到 JSON 端点 |
| **请求体大小限制 (X-LB)** | 发送超大 body |
| **SQLite 并发写入 (X-SQ)** | 10 个并发写请求 |
| **AI 供应商故障转移 (X-AF)** | 通过 API 设置主供应商 circuit_open=1，再请求 |

### 阶段 6：输出报告

输出到 `test/api-test-results/{模块名}-api-test-results.md`。

## 文档结构

```markdown
# {模块名} 接口测试结果报告

## 文档信息
| 项 | 值 |
|----|-----|
| 文档ID | ATR-{模块缩写}-{YYYYMMDD}-{序号} |
| 源文档ID | ATC-{模块缩写}-{YYYYMMDD}-{序号} |
| 源文档路径 | test/api-test-cases/formulas-api-test-cases.md |
| 源文件hash | {hash} |
| 执行时间 | YYYY-MM-DD HH:mm |
| 总用例数 | N |
| 通过 | N |
| 失败 | N |
| 跳过 | N |
| 通过率 | N% |
| 平均响应时间 | Nms |

## 执行结果总览
| 用例ID | 用例名称 | 结果 | 状态码 | 响应时间 |
|--------|---------|------|--------|---------|
| A01-P01 | 正常创建 | ✅ | 201 | 45ms |
| A01-E01 | 重复创建 | ✅ | 409 | 12ms |
| A01-B01 | name 为空 | ❌ | 500 | 8ms |
| A01-R01 | 未登录 | ✅ | 401 | 5ms |

## 失败用例详情

### A01-B01 name 为空
| 项 | 值 |
|----|-----|
| 用例ID | A01-B01 |
| 严重程度 | High |
| 预期状态码 | 400 |
| 实际状态码 | 500 |
| 预期响应 | {error:"名称必填"} |
| 实际响应 | {error:"Internal Server Error", stack:"TypeError..."} |
| 修复建议 | 添加 name 参数校验，返回 400 而非 500；同时响应不应包含堆栈信息（X-SE 安全问题） |

## 性能异常用例
| 用例ID | 端点 | 响应时间 | 阈值 | 建议 |
|--------|------|---------|------|------|
| A02-P01 | GET /api/formulas | 3500ms | 1000ms | 检查是否缺少索引或 N+1 查询 |

## Bug 汇总（按严重程度排序）
| 用例ID | Bug 描述 | 严重程度 | 修复建议 |
|--------|---------|---------|---------|
| A01-B01 | name 为空返回 500 而非 400 | High | 添加参数校验 |
| A01-B01 | 500 错误泄露堆栈信息 | High | 生产环境关闭错误详情 |
| A02-P01 | 列表查询响应 3.5s | Medium | 添加索引或优化查询 |
```

## 严重程度判定规则

| 条件 | 严重程度 |
|------|---------|
| 数据丢失/损坏/安全漏洞 | Critical |
| 核心接口不可用/返回错误状态码 | High |
| 非核心接口异常/性能不达标 | Medium |
| 响应格式不规范/不影响功能 | Low |

## 测试数据管理

| 策略 | 说明 |
|------|------|
| 测试前 | 清理残留测试数据 + seed 基础数据 |
| 测试中 | 每条用例独立，不依赖其他用例的副作用 |
| 测试后 | 保留测试数据供排查，或按配置清理 |
| 命名规范 | 测试数据以 `[test]` 前缀标识，便于清理 |

## 注意事项

- **用例ID 透传**：结果文档中的用例ID 必须与 api-test-cases 文档完全一致
- **文档ID 互引**：结果文档头部必须包含源文档ID 和源文件hash
- **测试数据隔离**：每条用例独立准备数据，不依赖执行顺序
- **不修改生产数据**：测试数据必须可辨识（[test] 前缀），测试后可清理
- **性能阈值可配置**：不同端点类型有不同阈值，超时不算失败但标记为性能异常
- **并发测试安全**：并发用例使用独立的测试数据，避免相互干扰
