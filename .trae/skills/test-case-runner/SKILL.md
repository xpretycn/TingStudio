# 测试用例执行器

读取测试用例文档，用 Playwright 逐条执行，输出测试结果报告。失败的用例自动截图，供 bug-fixer 修复。

## 触发词

跑测试用例、执行测试、运行测试、跑测试

## 与上游的关联

本技能读取 `ui-test-case-generator` 生成的测试用例文档，通过**文档ID**和**用例ID**实现全链路追溯。

| 关联字段 | 来源 | 用途 |
|---------|------|------|
| 文档ID | test-cases 文档头部 | 唯一标识，写入结果文档头部 |
| 用例ID | test-cases 中每条用例 | 直接透传，不做二次编号 |
| 文件hash | test-cases 文档头部 | 校验用例文档是否变更 |

## 执行流程

```
读取测试用例 → 变更校验 → 启动 Playwright → 逐条执行 → 记录结果 → 输出报告
```

### 阶段 1：读取测试用例

1. 确定目标测试用例文档：
   - 用户指定了文档路径 → 直接使用
   - 用户指定了页面名称 → 查找 `test/test-cases/{页面名}-test-cases.md`
   - 用户未指定 → 扫描 `test/test-cases/` 目录，列出所有文档供选择

2. 解析文档，提取：
   - 文档ID（如 `TC-FF-20260525-001`）
   - 文件hash
   - 所有交互元素及其用例（ID、前置条件、操作步骤、预期结果）

### 阶段 2：变更校验

1. 计算当前 test-cases 文件的实际 hash
2. 与文档头部的文件hash 对比：
   - **hash 一致** → 正常执行
   - **hash 不一致** → 警告"测试用例文档已变更，建议先重新生成"，用户确认后继续

3. 检查是否已有对应的 test-results 文档：
   - 已有 → 增量模式：只执行新增/未执行的用例，保留已有结果
   - 无 → 全量模式：执行所有用例

### 阶段 3：启动 Playwright

1. 检查本地开发服务器是否运行（默认 `http://localhost:5173`）
2. 未运行 → 启动 `npm run dev`（前端）
3. 等待服务器就绪
4. 初始化 Playwright 浏览器实例

### 阶段 4：逐条执行

对每条用例：

1. **准备前置条件**：导航到目标页面，设置必要的状态（登录、填入依赖数据）
2. **执行操作步骤**：按步骤模拟用户操作（点击、输入、选择等）
3. **验证预期结果**：对比实际页面状态与预期结果
4. **记录结果**：
   - 通过 ✅ → 记录"通过"
   - 失败 ❌ → 截图保存到 `test/screenshots/{用例ID}.png`，记录实际结果
   - 跳过 ⏭ → 记录跳过原因

#### 多层验证策略

对正向流程和联动功能用例，按以下层次验证：

| 验证层 | 验证方式 | 适用用例 |
|--------|---------|---------|
| **UI 层** | 检查元素可见性、文字内容、样式状态 | 所有用例 |
| **网络层** | 拦截 API 请求，验证请求路径、参数、状态码 | 涉及提交/保存/删除的用例 |
| **数据层** | 刷新页面后验证数据持久化 | 涉及增删改的用例 |
| **计算层** | 提取页面计算结果，与预期值对比（误差 < 0.01） | X-C 类用例（营养成分、利润率等） |
| **路由层** | 刷新/后退后验证 URL 和页面状态 | X-R 类用例 |

#### 特殊场景执行策略

| 场景 | 执行方式 |
|------|---------|
| **权限角色 (R)** | 切换不同角色账号登录，验证按钮可见性和操作权限 |
| **状态流转 (S)** | 先通过 API 将数据设为目标状态，再验证操作是否受限 |
| **计算准确性 (X-C)** | 填入已知数据，提取页面计算结果，与手动计算值对比 |
| **并发冲突 (X-CC)** | 开两个浏览器 Context，同时操作同一数据，验证冲突处理 |
| **浏览器路由 (X-R)** | 操作后刷新页面或浏览器后退，验证状态保持 |
| **搜索筛选组合 (X-F)** | 组合多个筛选条件+排序+翻页，验证结果正确性和条件保持 |
| **版本化 (X-V)** | 通过 API 创建多版本数据，验证版本对比/回退/查询行为 |
| **AI 功能 (X-AI)** | 模拟 AI 响应（mock API），验证降级/超时/上下文保持 |
| **导出下载 (X-EX)** | 触发导出后下载文件，验证文件内容；访问过期分享链接 |
| **文件解析 (X-FP)** | 上传已知文件验证缓存命中；上传错误格式验证拦截 |
| **主题切换 (X-T)** | 切换品牌色/亮暗模式后遍历页面组件，验证颜色跟随和对比度 |
| **登录会话 (X-AU)** | 手动设置 Token 过期时间，验证跳转；双 Context 模拟并发登录 |
| **异步任务 (X-AS)** | 触发异步任务后轮询进度，验证进度更新；中途取消验证清理 |
| **数据一致性 (X-DC)** | 通过 API 部分失败场景验证回滚；删除后查询关联表验证级联 |

**无法自动执行的用例处理**：

| 情况 | 处理方式 |
|------|---------|
| 需要断网模拟 | 标记"需手动验证"，跳过自动执行 |
| 需要特定权限账号 | 标记"需手动验证"，跳过自动执行 |
| 涉及第三方服务（支付、短信） | 标记"需手动验证"，跳过自动执行 |
| Playwright 无法操作的组件 | 标记"自动执行失败"，附原因 |
| 前置条件无法自动满足 | 标记"前置条件不满足"，跳过 |

### 阶段 4.5：视觉回归测试

在 E2E 功能测试完成后，对每个页面执行视觉回归检测：

1. **基准截图管理**：
   - 首次执行：对每个页面状态（默认/品牌色切换/亮暗模式）截图，保存到 `test/baselines/{页面名}/` 作为基准
   - 后续执行：截图后与基准对比，差异超过阈值标记为失败

2. **截图范围**：

| 截图场景 | 说明 | 存储路径 |
|---------|------|---------|
| 默认主题 | 当前品牌色 + 亮色模式 | `test/baselines/{页面名}/default.png` |
| 品牌色切换 | pink/blue/green/yellow 各一张 | `test/baselines/{页面名}/brand-{color}.png` |
| 暗色模式 | 当前品牌色 + 暗色模式 | `test/baselines/{页面名}/dark.png` |
| 关键交互态 | 表单校验错误、loading、空状态等 | `test/baselines/{页面名}/{用例ID}.png` |

3. **对比规则**：
   - 使用 Playwright 的 `toHaveScreenshot()` 或像素对比
   - 差异阈值：0.1%（允许微小渲染差异）
   - 差异超过阈值 → 生成差异图（红色高亮变化区域），保存到 `test/screenshots/diff/{页面名}/`

4. **视觉回归结果**：
   - 与基准一致 → ✅ 通过
   - 差异在阈值内 → ✅ 通过（附差异百分比）
   - 差异超阈值 → ❌ 失败，附差异图和基准图路径
   - 无基准图 → 🆕 新基准，提示用户确认

5. **基准更新**：
   - 用户确认视觉变更是预期行为后，更新基准截图
   - 命令："更新视觉基准" → 当前截图覆盖基准

### 阶段 5：输出报告

输出到 `test/test-results/{页面名}-test-results.md`。

## 严重程度判定规则

| 条件 | 严重程度 |
|------|---------|
| 页面崩溃/白屏/数据丢失 | Critical |
| 核心功能不可用/阻塞主流程 | High |
| 非核心功能异常/有替代路径 | Medium |
| UI 展示问题/不影响功能 | Low |

## 注意事项

- **用例ID 透传**：结果文档中的用例ID 必须与 test-cases 文档完全一致，不做二次编号
- **文档ID 互引**：结果文档头部必须包含源文档ID 和源文件hash，供 bug-fixer 追溯
- **截图命名**：统一用用例ID 命名（如 `E01-E01.png`），便于定位
- **增量执行**：已有结果文档时，只执行新增/未执行的用例，保留已有结果
- **服务器管理**：执行完毕后不自动关闭开发服务器，避免影响用户
- **遵循 playwright-no-npx 规则**：使用全局安装的 Playwright，不用 npx 调用
