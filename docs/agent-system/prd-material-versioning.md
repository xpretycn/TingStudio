# PRD：原料管理版本化系统

## 1. Executive Summary

### Problem Statement

原料是配方的基础数据，当前系统中所有原料可被任意编辑，一旦原料营养数据变更，引用该原料的所有配方营养成分将产生偏差甚至失效。系统不设审批流程，因此无法通过人工审核来管控原料变更。

### Proposed Solution

引入版本化机制 + 用户数据域隔离：原料一旦被配方引用即不可原地编辑，修改操作自动创建新版本（新原料记录）；配方中冗余存储引用时刻的营养快照，确保旧配方不受影响。同时，配方师只能编辑自己创建的原料，管理员可管理全局。

### Success Criteria

1. 配方师可自由新建带新原料的配方，无需等待审批
2. 已有配方的营养数据不受原料变更影响
3. 管理员可在全局视角管理所有原料版本
4. 原料变更历史可追溯，支持版本查看

## 2. User Experience & Functionality

### User Personas

| 角色 | 描述 |
|------|------|
| **管理员 (admin)** | 管理全局原料数据，可编辑/删除任何原料，查看所有版本 |
| **配方师 (formulist)** | 创建和管理自己的原料，可使用其他配方师的原料但不能修改 |

### User Stories

| ID | 用户故事 | 验收条件 |
|:--:|----------|----------|
| US-01 | 作为**配方师**，我想**新建一个原料**，以便**在新配方中使用该原料** | ① 新建原料时填写名称、类别、营养数据 ② 创建成功后原料立即可用 ③ 其他用户可查看但不可编辑 |
| US-02 | 作为**管理员**，我想**编辑一个已被引用的原料**，以便**纠正错误的营养数据** | ① 编辑时自动创建新版本 ② 旧版本保留完整数据 ③ 系统提示将生成新版本 |
| US-03 | 作为**配方师**，我想**修改自己创建的原料**，以便**更新数据** | ① 只能编辑自己创建的原料 ② 若已被引用，自动创建新版本 ③ 未被引用时允许原地编辑 |
| US-04 | 作为**用户**，我想**查看原料的版本历史**，以便**了解数据变更过程** | ① 展示版本列表（版本号+时间+操作人） ② 可点击查看任一版本详情 ③ 版本间差异高亮显示 |
| US-05 | 作为**配方师**，我在**创建配方时引用原料**，配方的**营养计算不受原料后续变更影响** | ① 配方保存时自动快照原料营养数据 ② 原料变更后配方营养不变 ③ 配方中显示原料版本信息 |

### Non-Goals

- 不实现审批工作流
- 不做原料的"发布/草稿"状态管理
- 不做原料版本自动合并
- 不做原料数据的批量导入版本控制

## 3. Technical Specifications

### Architecture Overview

```
┌─────────────────────────────────────────────────┐
│                  前端 (Vue 3)                     │
│  MaterialList.vue  ──  MaterialVersion.vue       │
│       │                    │                     │
│       └────┬───────────────┘                     │
│            ▼                                     │
│       materialStore (Pinia)                      │
└────────────┼─────────────────────────────────────┘
             │ HTTP /api/materials/*
┌────────────┼─────────────────────────────────────┐
│            ▼                                     │
│       routes / controllers                       │
│            │                                     │
│            ▼                                     │
│       materialService                            │
│            │                                     │
│            ▼                                     │
│   materials 表 ← history → versions 表           │
│   formulas 表 → formula_materials 快照           │
└──────────────────────────────────────────────────┘
```

### Versioning Mechanism

1. 原料表 `materials` 新增 `version`、`previous_version_id` 字段
2. 编辑已引用原料：INSERT 新行（新 UUID），设置 `previous_version_id` 指向旧版本
3. 编辑未引用原料：原地 UPDATE（无需创建新版本）
4. 删除：标记 `is_deleted = true`，保留历史数据

### Snapshot in Formula

- `formula_materials` 表存储引用时刻的原料营养快照
- 包含：原料 ID、版本号、引用时的营养数据（per100g 各字段）
- 营养计算使用快照数据，而非实时查询原料表

### Security & Privacy

- 编辑权限检查：`material.created_by` 对比当前用户 ID
- 管理员绕过权限检查
- 版本历史可追溯操作人

## 4. Risks & Roadmap

### Phased Rollout

| 阶段 | 内容 | 时间 |
|:----:|------|:----:|
| MVP | 版本化核心 + 用户域隔离 + 配方快照 | 当前 |
| v1.1 | 版本对比 + 版本列表 UI 增强 | 后续 |
| v2.0 | 回收站 + 批量引用更新 | 未来 |

### Technical Risks

| 风险 | 影响 | 缓解措施 |
|:----:|------|----------|
| 配方快照数据量增长 | 存储膨胀 | 仅快照营养相关字段，非全量原料数据 |
| 版本链过长 | 查询性能下降 | 限制显示最新 N 个版本，分页加载 |
| 已有数据迁移 | 数据不一致 | 迁移脚本：为已有原料设置 version=1，创建初始快照 |

## 6. 页面交互设计

### 6.1 原料列表页 - 版本展示

- 列表新增"版本"列：绿色标签显示最新版本（v2），灰色标签显示历史版本
- 点击版本号直接跳转版本历史页
- tooltip 悬浮显示："共 N 个版本"

### 6.2 权限控制下的操作按钮

- **admin**：所有原料显示 编辑·版本历史·删除
- **formulist 看自己的**：显示 编辑·版本历史
- **formulist 看别人的**：仅显示 版本历史（编辑按钮隐藏）

### 6.3 版本化编辑确认流

- 编辑已引用原料 → 弹出 Popconfirm 确认框（不是 Dialog）
- 显示：引用该原料的配方列表 + 版本化后果说明
- 确认后→编辑表单进入"版本化模式"（顶部提示条）
- 保存按钮文案变为"保存并创建新版本"

### 6.4 版本历史页（全新）

- 时间线布局：`● v2 → ● v1`（自上而下）
- 每条显示：版本号、时间、操作人、变更摘要
- 点击展开版本详情（侧边栏或行内展开）
- 空状态："此原料尚无版本变更历史"

### 6.5 配方中的版本展示

- 配方原料表每行显示原料版本号
- 最新版本→绿色「最新」标签
- 非最新版本→黄色「历史」标签
- 原料有新版本时提示"是否刷新快照"

## 7. 核心约束

- ❌ 不设审批流程
- ✅ 修改已引用原料自动创建新版本
- ✅ 旧配方始终引用旧版本数据
- ✅ 可创建新配方引用新版本
- ✅ 用户域隔离（配方师只能编辑自己的原料）
- ✅ 管理员可管理全局
