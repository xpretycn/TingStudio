## FloatDrawer和FloatBubble 组件设计方案

### 1. 核心设计原则

- **上下文感知**：关联当前页面/操作的业务数据（配方/原料/报表）；

### 2. 组件结构（分层设计）

#### （1）基础悬浮层（常驻态）

| 元素         | 功能说明                                                                  |
| ------------ | ------------------------------------------------------------------------- |
| 未读消息提示 | 红点/数字角标，关联 `agent_messages` 未读消息数                           |
| 快捷功能入口 | 悬停显示 3 个核心快捷按钮：「含量比校验」「配方建议」「报价单」           |
| 状态指示灯   | 绿色（在线）/黄色（加载中）/红色（模型异常），关联 `ai_health_records` 表 |

#### （2）对话交互层（展开态）

| 模块       | 核心功能                                                       | 关联数据库表                          |
| ---------- | -------------------------------------------------------------- | ------------------------------------- |
| 会话头部   | 会话标题（自动生成：如「配方 FP001 营养分析」）关闭/最小化按钮 | <br />                                |
| 消息列表   | 支持文字消息，区分用户/AI 消息，加载态动效                     | `agent_messages`（content/type/role） |
| 输入区     | 文字输入框 + 发送按钮                                          | <br />                                |
| 功能扩展栏 | 左侧/底部显示「指令模板」                                      | `agent_role_config`（角色配置）       |

#### （3）操作执行层（交互延伸态）

| 功能模块 | 执行逻辑                                                                                                   |
| -------- | ---------------------------------------------------------------------------------------------------------- |
| 指令解析 | AI 识别用户指令（如「计算配方 FP001 的营养成分」），解析为业务操作指令                                     |
| 确认弹窗 | 高危操作（如修改配方、批量导出）触发确认，关联 `agent_pending_confirmations`                               |
| 进度反馈 | 显示操作进度（如「营养报告生成中 60%」），关联 `export_jobs`/`nutrition_analysis_reports` 的 progress 字段 |
| 结果展示 | 操作完成后展示结果（如报告预览、导出链接），支持一键跳转至对应业务页面                                     |

### 3. 核心功能设计

#### （1）上下文感知能力

- **页面级上下文**：通过路由/页面状态获取当前业务对象 ID（如配方 ID、原料 ID），对话初始化时自动携带；
  示例：用户在「配方 FP001 编辑页」唤起助手，自动发送上下文：`当前操作：配方 FP001（ID：xxx）编辑，原料列表：[{"materialId":"xxx","name":"白砂糖"}]`；
- **权限上下文**：读取当前用户 `users` 表的 `role` 字段，限制助手功能（如配方师不可调用管理员级别的销量数据）。

#### （2）核心业务能力映射

| AI 助手功能           | 底层业务逻辑                                                                        | 关联 API/表操作                                                               |
| --------------------- | ----------------------------------------------------------------------------------- | ----------------------------------------------------------------------------- |
| 配方原料搭配建议      | 基于 `materials` 表原料类型/营养成分，结合 AI 模型推荐搭配方案                      | 读取 `materials`/`material_nutrition`，生成建议后支持一键添加至 `formulas` 表 |
| 营养成分计算/合规校验 | 调用营养计算逻辑，对比 `nutrition_profiles` 标准，生成 `nutrition_analysis_reports` | 创建 `formula_nutrition_summaries` 记录，生成合规报告                         |
| 配方销量数据统计分析  | 聚合 `formula_sales` 表数据，按周期/业务员维度生成统计结果                          | 执行 `SELECT` 聚合查询，返回可视化数据（支持导出为 `reports` 表记录）         |
| 导出模板推荐/生成     | 基于 `export_templates` 表历史模板，推荐适配当前配方的导出配置                      | 创建 `export_jobs` 任务，返回文件下载链接                                     |

#### （3）异常处理与降级

- **模型异常**：读取 `ai_health_records`/`ai_alert_records` 表，模型不可用时切换至 `ai_fallback_configs` 配置的降级方案（预设回复）；
- **权限不足**：检测当前用户 `users` 表的 `role`，拒绝执行越权操作并提示；
- **数据缺失**：如配方无原料数据时，助手提示「请先完善配方原料信息」，并引导跳转编辑页。

### 4. 技术实现要点

#### 1）后端适配

- **消息处理**：新增 `POST /api/agent/message` 接口，接收用户消息，调用 AI 模型，存储至 `agent_messages` 表；
- **上下文解析**：新增上下文解析中间件，提取请求中的业务对象 ID，关联对应表数据；
- **操作执行**：新增 `POST /api/agent/action` 接口，执行 AI 解析后的业务操作（如生成营养报告），并更新对应表（`nutrition_analysis_reports`/`export_jobs`）。

#### （2）数据存储适配

- 扩展 `agent_messages` 表：新增 `business_object_id`/`business_object_type` 字段，关联消息对应的业务对象（配方/原料/报表）；
- 利用 `ai_usage_logs` 表：记录悬浮助手的模型调用量。
