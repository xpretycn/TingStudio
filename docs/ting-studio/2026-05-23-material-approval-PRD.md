## 1. 概述

### 1.1 问题陈述

当前 TingStudio 原料管理存在以下核心缺陷：

1. **无审批流程**：原料创建后立即可用，任何 formulist 均可自由创建和编辑原料，缺少质量把关环节，可能导致营养数据错误或不完整的原料被配方引用。
2. **无状态管理**：原料没有"发布/草稿"状态区分，所有原料均可被直接编辑和使用，无法区分哪些原料已经过审核、哪些仍在编辑中。
3. **与配方模块不对齐**：配方模块已具备完整的审批工作流（draft → pending_review → published → archived），但原料模块缺少对等机制，形成管理盲区。

### 1.2 解决方案

在现有 `materials` 表版本化体系上，新增审批工作流和发布/草稿状态管理功能，完全对齐配方模块的审批机制，形成原料全生命周期管理闭环。

### 1.3 成功指标

| 指标 | 目标值 | 衡量方式 |
|------|--------|----------|
| 审批流程覆盖率 | 100% 原料发布必须经过审批 | 线上无未经审批的发布记录 |
| 状态转换准确率 | 100% 状态转换符合业务规则 | 状态变更日志审计 |
| 审批日志完整性 | 每次状态变更均有日志记录 | 日志覆盖率检查 |
| 仪表盘集成 | 原料审批与配方审批统一展示 | UI 验收 |
| 数据迁移一致性 | 现有原料全部正确迁移为 published | 迁移后数据校验 |

## 2. 用户角色与权限

### 2.1 角色定义

| 角色 | 说明 | 系统字段 |
|------|------|----------|
| admin | 管理员，拥有全部权限 | `users.role = 'admin'` |
| formulist | 配方师，仅可操作自己创建的数据 | `users.role = 'formulist'` |

### 2.2 权限矩阵

| 操作 | admin | formulist | 权限检查方式 |
|------|-------|-----------|-------------|
| 创建草稿原料 | YES | YES | `authMiddleware` |
| 编辑草稿原料 | YES | YES（仅自己创建的） | `authMiddleware` + `created_by` 校验 |
| 提交审批（draft → pending_review） | YES | YES（仅自己创建的） | `authMiddleware` + `created_by` 校验 |
| 审批通过（pending_review → published） | YES | NO | `authMiddleware` + `role === 'admin'` |
| 审批驳回（pending_review → draft） | YES | NO | `authMiddleware` + `role === 'admin'` |
| 直接发布（draft → published） | YES | NO | `authMiddleware` + `role === 'admin'` |
| 编辑已发布原料 | YES（创建新版本，status=draft） | YES（仅自己创建的，创建新版本） | `authMiddleware` + `created_by` 校验 |
| 删除原料 | YES（软删除） | NO | `authMiddleware` + `role === 'admin'` |
| 查看待审批列表 | YES | NO | `authMiddleware` + `role === 'admin'` |
| 查看审批日志 | YES | YES | `authMiddleware` |

### 2.3 数据隔离

- admin 可查看所有原料
- formulist 可查看所有原料列表，但只能编辑/提交自己创建的原料
- 审批操作不受数据隔离限制（admin 可审批任何人的原料）

## 3. 用户故事

### 3.1 审批流程

#### US-1.1：配方师提交原料审批

> 作为配方师，我希望将草稿状态的原料提交审批，以便管理员审核后正式发布。

**验收标准**：
- AC-1.1.1：草稿原料（`status = 'draft'`）显示"提交审批"按钮
- AC-1.1.2：点击后状态变为 `pending_review`，按钮消失
- AC-1.1.3：待审批状态的原料不可编辑
- AC-1.1.4：仅原料创建人或 admin 可提交审批
- AC-1.1.5：提交时可选填提交说明

#### US-1.2：管理员审批通过

> 作为管理员，我希望审批通过待审核的原料，使其成为正式发布状态。

**验收标准**：
- AC-1.2.1：审批通过后原料状态变为 `published`
- AC-1.2.2：审批通过时可选填审批意见
- AC-1.2.3：仅 admin 角色可执行审批通过操作
- AC-1.2.4：审批操作记录到 `material_review_logs` 表

#### US-1.3：管理员审批驳回

> 作为管理员，我希望驳回不符合要求的原料，并注明驳回原因，以便配方师修改后重新提交。

**验收标准**：
- AC-1.3.1：驳回后原料状态回退为 `draft`
- AC-1.3.2：驳回时**必须**填写驳回原因（最少 5 个字符）
- AC-1.3.3：配方师在原料详情中可查看驳回原因
- AC-1.3.4：驳回后配方师可修改并重新提交审批
- AC-1.3.5：仅 admin 角色可执行驳回操作

#### US-1.4：管理员直接发布

> 作为管理员，我希望可以跳过审批流程直接发布原料，以便紧急情况下快速上线。

**验收标准**：
- AC-1.4.1：仅 admin 可执行直接发布操作
- AC-1.4.2：草稿或待审批状态的原料可直接发布
- AC-1.4.3：直接发布操作记录到审批日志

#### US-1.5：管理员审批列表

> 作为管理员，我希望在一个统一的页面查看所有待审批的原料和配方，以便高效处理审批任务。

**验收标准**：
- AC-1.5.1：审批列表展示所有 `status = 'pending_review'` 的原料
- AC-1.5.2：支持按原料名称、提交人筛选
- AC-1.5.3：列表显示：原料名称、编码、类型、提交人、提交时间
- AC-1.5.4：每行提供"通过"、"驳回"操作
- AC-1.5.5：与配方审批集成在同一面板

### 3.2 发布/草稿状态管理

#### US-2.1：创建草稿原料

> 作为配方师，我希望新建的原料默认为草稿状态，以便在完成所有信息填写后再提交发布。

**验收标准**：
- AC-2.1.1：新建原料的默认状态为 `draft`
- AC-2.1.2：草稿状态的原料可自由编辑
- AC-2.1.3：草稿状态的原料不出现在配方的原料选择列表中（仅 published 原料可选）
- AC-2.1.4：草稿状态的原料在列表中有明显的"草稿"标识

#### US-2.2：编辑已发布原料

> 作为配方师/管理员，我希望编辑已发布原料时自动创建新版本并重置为草稿状态，以便变更经过审批后再生效。

**验收标准**：
- AC-2.2.1：编辑已发布原料时，系统自动创建新版本（版本号递增）
- AC-2.2.2：新版本的状态为 `draft`，需走审批流程
- AC-2.2.3：原发布版本保持 `published` 状态不变
- AC-2.2.4：编辑前弹出确认提示，说明将创建新版本
- AC-2.2.5：新版本创建后，`is_latest` 标记转移到新版本

#### US-2.3：原料状态筛选

> 作为用户，我希望在原料列表中按状态筛选原料，以便快速找到草稿或待审批的原料。

**验收标准**：
- AC-2.3.1：列表页支持按状态筛选：全部/草稿/待审批/已发布
- AC-2.3.2：默认显示已发布的原料
- AC-2.3.3：每个状态的数量在筛选标签旁显示

### 3.3 审批日志

#### US-3.1：查看审批日志

> 作为用户，我希望查看原料的审批历史记录，以便了解原料的审核过程。

**验收标准**：
- AC-3.1.1：原料详情页展示审批日志时间线
- AC-3.1.2：每条日志显示：操作类型、操作人、操作时间、审批意见
- AC-3.1.3：操作类型包括：提交审批、审批通过、审批驳回、直接发布
- AC-3.1.4：日志按时间倒序排列

## 4. 功能需求

### 4.1 Feature 1：原料审批流程

#### 4.1.1 状态机设计

**新状态机**：

```
                    ┌──────────────────────────────┐
                    │       admin 直接发布           │
                    │  (publishMaterial)             │
                    ▼                               │
  ┌────────┐  submit   ┌──────────────┐  approve   ┌───────────┐
  │ draft  │ ────────► │pending_review│ ─────────► │ published │
  └────────┘           └──────────────┘            └───────────┘
       ▲                     │                           │
       │                     │ reject                    │ 编辑时创建新版本
       │                     ▼                           │ (status=draft)
       └─────────────── draft ◄──────────────────────────┘
                  (驳回回退为草稿)              重新编辑
```

**状态定义**：

| 状态 | 值 | 说明 | 可执行操作 |
|------|-----|------|-----------|
| 草稿 | `draft` | 初始状态，可编辑 | 提交审批、编辑、删除 |
| 待审批 | `pending_review` | 已提交等待审核 | 管理员审批通过/驳回、管理员直接发布 |
| 已发布 | `published` | 审批通过正式发布 | 编辑（创建新版本）、删除 |

#### 4.1.2 数据库变更

**`materials` 表新增字段**：

```sql
ALTER TABLE materials ADD COLUMN status TEXT NOT NULL DEFAULT 'draft'
  CHECK(status IN ('draft', 'pending_review', 'published'));
```

**新增 `material_review_logs` 表**：

```sql
CREATE TABLE IF NOT EXISTS material_review_logs (
  review_log_id TEXT PRIMARY KEY,
  material_id TEXT NOT NULL,
  reviewer_id TEXT NOT NULL,
  reviewer_name TEXT DEFAULT NULL,
  action TEXT NOT NULL CHECK(action IN ('submit', 'approve', 'reject', 'publish')),
  comment TEXT DEFAULT NULL,
  created_at TEXT NOT NULL DEFAULT (datetime('now')),
  FOREIGN KEY (material_id) REFERENCES materials(id) ON DELETE CASCADE,
  FOREIGN KEY (reviewer_id) REFERENCES users(id) ON DELETE SET NULL
);
```

#### 4.1.3 API 设计

| 方法 | 路径 | 说明 | 权限 |
|------|------|------|------|
| POST | `/api/materials/:id/submit-review` | 提交审批 | 创建人或 admin |
| PUT | `/api/materials/:id/approve` | 审批通过 | admin |
| PUT | `/api/materials/:id/reject` | 审批驳回 | admin |
| PUT | `/api/materials/:id/publish` | 直接发布 | admin |
| GET | `/api/materials/pending-review` | 待审批列表 | admin |
| GET | `/api/materials/:id/review-logs` | 审批日志 | 认证用户 |

### 4.2 Feature 2：发布/草稿状态管理

#### 4.2.1 创建流程变更

- 新建原料默认 `status = 'draft'`
- 草稿原料不出现在配方的原料选择下拉列表中
- 草稿原料在列表中有"草稿"状态标签

#### 4.2.2 编辑流程变更

- 草稿原料：直接编辑（原地 UPDATE）
- 待审批原料：不可编辑
- 已发布原料：编辑时创建新版本，新版本 `status = 'draft'`

#### 4.2.3 列表筛选增强

- 新增 `status` 查询参数
- 默认仅显示 `published` 原料（配方选择场景）
- 管理页面可切换显示全部状态

### 4.3 Feature 3：审批日志

#### 4.3.1 日志记录范围

| 操作 | action 值 | 触发场景 |
|------|-----------|---------|
| 提交审批 | `submit` | draft → pending_review |
| 审批通过 | `approve` | pending_review → published |
| 审批驳回 | `reject` | pending_review → draft |
| 直接发布 | `publish` | draft/pending_review → published |

#### 4.3.2 日志字段

| 字段 | 说明 |
|------|------|
| `review_log_id` | 日志唯一 ID |
| `material_id` | 关联原料 ID |
| `reviewer_id` | 操作人 ID |
| `reviewer_name` | 操作人姓名 |
| `action` | 操作类型 |
| `comment` | 审批意见 |
| `created_at` | 操作时间 |

### 4.4 Feature 4：仪表盘集成

#### 4.4.1 审批卡片增强

在现有 `ApprovalCard.vue` 中增加原料审批 Tab：

- **admin 视角**：待审核 Tab 中同时展示配方和原料的待审批记录
- **formulist 视角**：我的提交 Tab 中同时展示配方和原料的提交记录

#### 4.4.2 审批面板增强

- `AdminReviewPanel.vue`：增加"原料审批"分区或 Tab
- `MyApprovalPanel.vue`：增加"原料提交"分区或 Tab

## 5. 非功能需求

### 5.1 性能要求

| 场景 | 指标 |
|------|------|
| 审批列表加载 | <= 1s（100 条以内） |
| 状态变更操作 | <= 500ms |
| 审批日志查询 | <= 500ms |

### 5.2 数据一致性

| 约束 | 说明 |
|------|------|
| 状态转换合法性 | 仅允许定义的状态转换路径 |
| 审批日志不可变 | 已创建的审批日志不可修改或删除 |
| 版本与状态一致 | 新版本创建时 status 必须为 draft |

### 5.3 安全要求

| 要求 | 实现方式 |
|------|---------|
| 权限校验 | 所有接口使用 `authMiddleware`，审批/驳回额外校验 admin 角色 |
| 数据隔离 | formulist 仅可操作自己创建的原料 |
| 输入校验 | comment 长度限制，reject 时 comment 必填 |
| SQL 注入防护 | 所有查询使用参数化查询 |

### 5.4 兼容性

| 项目 | 要求 |
|------|------|
| 现有版本化机制 | 审批状态与版本化字段（version, previous_version_id, is_latest）兼容 |
| 现有 API | 原有 CRUD 接口保持向后兼容，增加 status 相关逻辑 |
| 数据迁移 | 现有原料全部设为 published，不影响已有配方引用 |

## 7. 风险与缓解

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|---------|
| 现有原料数据迁移 | 已有原料无 status 字段 | 高 | 迁移脚本将所有现有原料设为 published |
| 草稿原料对配方选择的影响 | 配方师找不到新建的原料 | 中 | 列表页明确区分状态，提供筛选 |
| 审批流程增加发布周期 | 原料上线速度变慢 | 高 | admin 可直接发布跳过审批 |
| 与版本化机制冲突 | 状态与版本逻辑交叉 | 中 | 明确规则：新版本始终为 draft，发布后旧版本保持 published |

## 8. 实施路线

### Phase 1：数据库迁移与后端核心（优先级：高）

1. 数据库迁移：materials 表新增 status 字段 + 创建 material_review_logs 表
2. 后端：审批相关接口（submit/approve/reject/publish/review-logs）
3. 后端：状态管理逻辑（创建默认 draft、编辑已发布创建新版本）

### Phase 2：前端状态管理（优先级：高）

1. 原料列表页：状态标签、状态筛选
2. 原料表单页：草稿/审批状态提示、提交审批按钮
3. 原料详情页：审批日志时间线

### Phase 3：仪表盘集成（优先级：中）

1. ApprovalCard 集成原料审批
2. AdminReviewPanel 增加原料审批区
3. MyApprovalPanel 增加原料提交追踪

### Phase 4：测试与优化（优先级：中）

1. 审批流程端到端测试
2. 状态管理边界测试
3. 数据迁移验证

## 9. 非目标（本次不实现）

| 功能 | 说明 | 原因 |
|------|------|------|
| 原料审批撤回 | 提交后不可撤回 | 需 admin 驳回，流程更清晰 |
| 批量审批 | 同时审批多个原料 | 优先保证单条审批流程稳定 |
| 审批通知 | 审批结果推送通知 | 需要消息推送基础设施 |
| 原料归档状态 | published → archived | 当前业务不需要归档原料 |
| 审批超时自动处理 | 审批超时自动通过/驳回 | 需要定时任务基础设施 |
