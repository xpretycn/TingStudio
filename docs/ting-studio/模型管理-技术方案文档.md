## 1. 方案选型

### 1.1 三种方案对比

| 维度 | 方案A：独立路由 | 方案B：工具箱集成 | 方案C：个人设置集成 |
|------|----------------|------------------|-------------------|
| 页面空间 | 独立全屏，扩展性强 | 受工具箱布局限制 | 设置页空间有限 |
| 导航可达性 | 侧边栏一级入口 | 二级入口 | 二级入口 |
| 功能扩展 | 图表/表格/弹窗自由布局 | 需适配工具箱卡片风格 | 不适合放图表 |
| 权限控制 | 路由级 meta.requiresAdmin | 组件内判断 | 组件内判断 |
| 开发成本 | 中 | 低 | 低 |

### 1.2 选定方案

**方案A：独立路由** — 模型管理功能包含图表、表格、配置表单等复杂交互，需要独立全屏空间，且作为管理功能适合一级入口。

### 2.2 数据流

1. **模型加载**：应用启动 → `AIService.reloadModels()` → 从 DB 读取模型配置 → 替换内存中的硬编码配置
2. **AI 调用**：用户请求 → `aiController` → `AIService.chatCompletion()` → 调用 API + 记录用量 + 检查预警 → 如失败则自动切换
3. **健康检测**：`ModelHealthChecker` 定时器 → 读取每个模型的检测间隔 → 跳过未到期的 → 对到期模型发送测试请求 → 更新健康状态
4. **前端展示**：`ModelManagement.vue` → `modelStore` → `modelApi` → 后端接口

## 4. 核心修改点

### 4.1 AIService.ts 改造

- 新增 `reloadModels()` 方法：从 DB 读取模型配置替换内存数组
- 新增 `getAvailableVersions(provider)` 方法：返回版本列表
- `chatCompletion()` 新增参数：`callType`、`userId`、`requestSummary`
- 新增 `recordUsage()` 方法：写入 `ai_usage_logs` 表
- 新增自动切换逻辑：主模型失败时尝试 `_fallbackProvider`

### 4.2 ModelHealthChecker.ts

- `start()` / `stop()` 生命周期管理
- `scheduleNext()` 每小时检查一次是否有模型需要检测
- `checkAll()` 遍历模型，根据 `health_check_interval_days` 判断是否到期
- `checkOne()` 对单个模型发送轻量请求，记录延迟和状态

### 4.3 数据库迁移

- 使用 `ensureTable()` 模式，首次启动自动建表
- `ensureInitialAiModels()` 将 .env 中的模型配置迁移到数据库
- 6 张新表：ai_models、ai_usage_logs、ai_alert_configs、ai_alert_records、ai_health_records、ai_fallback_configs

## 5. 关键技术决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 可视化库 | ECharts | 项目已引入，功能强大 |
| 健康检测调度 | setTimeout 递归 | 轻量级，无需引入 cron 库 |
| 模型版本列表 | 前端硬编码 | 版本列表变化不频繁，避免额外 API |
| 预警触发时机 | 用量记录时同步检查 | 实时性好，无需额外定时任务 |
| 自动切换 | AIService 内部处理 | 对上层透明，无需修改调用方 |
| 权限控制 | 路由 meta + 组件内 computed | 双重保障 |
