## 1. 概述

### 1.1 背景

本次导出中心优化采用**双轨制架构**，重新定义导出功能的交互模式与数据流向：

- **轨道一：业务页面就地导出** — 用户在 FormulaList.vue、MaterialList.vue 等业务页面直接调用 API 发起导出，无需跳转到导出中心。导出任务创建后，用户可继续在当前页面操作，导出完成后通过通知或下载入口获取文件。
- **轨道二：导出中心管理视图** — ExportCenter.vue（仅 admin 可访问）作为系统管理模块下的子模块，提供导出任务的查看、统计、重试、重新导出等管理能力，以及导出模板、分享链接、全局配置的管理。

在此架构下，**API 接口管理 Tab 被移除**。`api_data_interfaces` 表保留但标记为废弃（Deprecated），前端不再展示 API 接口管理 Tab，后端不再提供 api-interfaces 路由端点。表结构和数据保留用于数据完整性，未来如需恢复可基于此表重新实现。

> **v2.1 变更说明**：导出中心已移入系统管理模块，导出配置（`export_center_config`）的前端调用方从 SystemConfig.vue 变更为 ExportCenter.vue，数据库层面无表结构或字段变更。

当前数据库层面存在以下问题：

1. **模板分类缺失** — `export_templates` 无法区分配方模板、原料模板、周报模板、月报模板
2. **任务类型受限** — `export_jobs` 仅支持单配方导出（`formula_id NOT NULL`），无法支持原料批量导出、报表导出等场景
3. **MySQL/SQLite Schema 不一致** — `api_data_interfaces` 和 `share_configs` 两张表在 MySQL 和 SQLite 中的字段定义完全不同，导致生产环境功能异常
4. **缺少全局配置** — 导出中心没有统一的配置管理机制（默认格式、速率限制、文件命名规则等）
5. **必备字段缺失** — 部分表缺少 `updated_at` 等规范要求的字段
6. **API 接口管理已废弃** — `api_data_interfaces` 表对应的前后端功能已不再使用，需标记为废弃

### 1.2 设计目标

- 支持多数据类别导出（配方、原料、周报、月报）
- 统一 MySQL 与 SQLite 的 Schema 定义，以 SQLite 为权威来源
- 新增导出中心全局配置表
- 所有表满足项目数据库规范（`id`、`created_at`、`updated_at` 必备字段）
- 迁移脚本幂等可重复执行
- 明确标记 `api_data_interfaces` 为废弃，保留表结构用于数据完整性
- 导出中心定位为管理模块，`export_jobs.created_by` 用于区分不同用户的导出任务

### 1.3 数据类别枚举

```typescript
type DataCategory = "formula" | "material" | "weekly-report" | "monthly-report"
```

## 2. 现有表问题分析

### 2.1 export_templates

| 问题 | 说明 |
|------|------|
| 缺少 `category` 字段 | 无法按数据类别（配方/原料/周报/月报）分类模板 |
| 缺少 `updated_at` 字段 | 不符合项目数据库规范（必备字段） |

### 2.2 export_jobs

| 问题 | 说明 |
|------|------|
| `formula_id NOT NULL` | 并非所有导出都基于配方（原料导出、报表导出无配方 ID） |
| 缺少 `data_category` 字段 | 无法区分导出任务的数据类别 |
| 缺少 `target_ids_json` 字段 | 无法支持批量导出多个配方/原料 |
| 缺少 `period_start`/`period_end` | 报表导出需要时间范围参数 |
| `export_type` 包含 `'api'` | API 推送不属于文件导出，应从 CHECK 约束中移除 |
| 缺少 `updated_at` 字段 | 不符合项目数据库规范 |

### 2.3 api_data_interfaces（⚠️ 废弃）

| 问题 | 说明 |
|------|------|
| 功能已废弃 | 前端 ExportCenter.vue 已移除 API 接口管理 Tab |
| 路由已移除 | 后端已移除 `/api/exports/api-interfaces` 路由端点 |
| MySQL 不兼容 | MySQL 版本字段与 SQLite 版本完全不同，但已无活跃 CRUD 操作 |
| 数据保留 | 表结构和数据保留用于数据完整性，未来可能恢复 |

> **废弃原因**：双轨制架构下，导出中心定位为管理模块，不再承担 API 接口管理职能。API 推送功能在实际业务中未使用，维护成本高于业务价值。

### 2.4 share_configs

| 问题 | 说明 |
|------|------|
| MySQL 主键名不同 | MySQL 使用 `config_id`，SQLite 使用 `share_id`，控制器代码基于 `share_id` |
| MySQL 缺少核心字段 | MySQL 版本缺少 `password`、`allowed_emails_json`、`download_limit`、`download_count` |
| MySQL 字段名不同 | MySQL `expires_at` vs SQLite `expire_date` |
| MySQL 多余字段 | MySQL 版本有 `is_active` 字段，SQLite 无此字段 |

## 3. 表结构设计

### 3.1 export_templates（导出模板表）— 修改

#### 变更摘要

| 变更类型 | 字段 | 说明 |
|---------|------|------|
| 新增 | `category` | 数据类别分类 |
| 新增 | `updated_at` | 更新时间（规范必备） |

#### SQLite 完整建表语句

```sql
CREATE TABLE IF NOT EXISTS `export_templates` (
  `template_id` TEXT PRIMARY KEY,
  `name` TEXT NOT NULL,
  `description` TEXT DEFAULT NULL,
  `category` TEXT NOT NULL DEFAULT 'formula' CHECK(category IN ('formula', 'material', 'weekly-report', 'monthly-report')),
  `type` TEXT NOT NULL CHECK(type IN ('pdf', 'excel', 'print')),
  `format_config_json` TEXT NOT NULL,
  `is_default` INTEGER NOT NULL DEFAULT 0,
  `created_by` TEXT NOT NULL,
  `created_at` TEXT NOT NULL DEFAULT (datetime('now')),
  `updated_at` TEXT NOT NULL DEFAULT (datetime('now'))
);
```

#### MySQL 完整建表语句

```sql
CREATE TABLE IF NOT EXISTS `export_templates` (
  `template_id` VARCHAR(36) PRIMARY KEY,
  `name` VARCHAR(255) NOT NULL,
  `description` TEXT DEFAULT NULL,
  `category` VARCHAR(20) NOT NULL DEFAULT 'formula',
  `type` VARCHAR(20) NOT NULL,
  `format_config_json` JSON NOT NULL,
  `is_default` TINYINT(1) NOT NULL DEFAULT 0,
  `created_by` VARCHAR(36) NOT NULL,
  `created_at` TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  `updated_at` TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  INDEX `idx_et_category` (`category`),
  INDEX `idx_et_type` (`type`),
  INDEX `idx_et_category_type` (`category`, `type`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```

#### 字段说明

| 字段 | SQLite 类型 | MySQL 类型 | 约束 | 说明 |
|------|-----------|-----------|------|------|
| `template_id` | TEXT | VARCHAR(36) | PK | UUID，使用 `generateId()` 生成 |
| `name` | TEXT | VARCHAR(255) | NOT NULL | 模板名称 |
| `description` | TEXT | TEXT | 可空 | 模板描述 |
| `category` | TEXT | VARCHAR(20) | NOT NULL, DEFAULT 'formula' | 数据类别：formula / material / weekly-report / monthly-report |
| `type` | TEXT | VARCHAR(20) | NOT NULL | 导出类型：pdf / excel / print（移除 'api'） |
| `format_config_json` | TEXT | JSON | NOT NULL | 格式化配置 JSON |
| `is_default` | INTEGER | TINYINT(1) | NOT NULL, DEFAULT 0 | 是否为该类别+类型的默认模板 |
| `created_by` | TEXT | VARCHAR(36) | NOT NULL | 创建人用户 ID |
| `created_at` | TEXT | TIMESTAMP | NOT NULL, DEFAULT now | 创建时间 |
| `updated_at` | TEXT | TIMESTAMP | NOT NULL, DEFAULT now | 更新时间 |

#### `category` 枚举值说明

| 值 | 说明 | 典型模板 |
|----|------|---------|
| `formula` | 配方导出 | 配方标签 PDF、配方数据 Excel |
| `material` | 原料导出 | 原料清单 Excel、原料详情 PDF |
| `weekly-report` | 周报导出 | 周度汇总 Excel、周度分析 PDF |
| `monthly-report` | 月报导出 | 月度汇总 Excel、月度分析 PDF |

CREATE TABLE IF NOT EXISTS `export_jobs` (

#### MySQL 完整建表语句

```sql
CREATE TABLE IF NOT EXISTS `export_jobs` (
  `job_id` VARCHAR(36) PRIMARY KEY,
  `formula_id` VARCHAR(36) DEFAULT NULL,
  `version_id` VARCHAR(36) DEFAULT NULL,
  `template_id` VARCHAR(36) DEFAULT NULL,
  `data_category` VARCHAR(20) NOT NULL DEFAULT 'formula',
  `target_ids_json` JSON DEFAULT NULL,
  `export_type` VARCHAR(20) NOT NULL,
  `status` VARCHAR(20) NOT NULL DEFAULT 'pending',
  `file_url` VARCHAR(500) DEFAULT NULL,
  `file_name` VARCHAR(255) DEFAULT NULL,
  `progress` INT NOT NULL DEFAULT 0,
  `error_message` TEXT DEFAULT NULL,
  `period_start` VARCHAR(20) DEFAULT NULL,
  `period_end` VARCHAR(20) DEFAULT NULL,
  `created_by` VARCHAR(36) NOT NULL,
  `created_at` TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  `updated_at` TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  `completed_at` TIMESTAMP NULL DEFAULT NULL,

  INDEX `idx_ej_status` (`status`),

```

#### 字段说明

| 字段 | SQLite 类型 | MySQL 类型 | 约束 | 说明 |
|------|-----------|-----------|------|------|
| `job_id` | TEXT | VARCHAR(36) | PK | UUID，使用 `generateId()` 生成 |
| `formula_id` | TEXT | VARCHAR(36) | 可空 | 关联配方 ID，非配方导出时为 NULL |
| `version_id` | TEXT | VARCHAR(36) | 可空 | 关联配方版本 ID |
| `template_id` | TEXT | VARCHAR(36) | 可空 | 使用的导出模板 ID |
| `data_category` | TEXT | VARCHAR(20) | NOT NULL, DEFAULT 'formula' | 数据类别 |
| `target_ids_json` | TEXT | JSON | 可空 | 批量导出目标 ID 列表，JSON 数组格式，如 `["id1","id2"]` |
| `export_type` | TEXT | VARCHAR(20) | NOT NULL | 导出类型：pdf / excel（移除 'api'） |
| `status` | TEXT | VARCHAR(20) | NOT NULL, DEFAULT 'pending' | 任务状态：pending / processing / completed / failed |
| `file_url` | TEXT | VARCHAR(500) | 可空 | 导出文件访问路径 |
| `file_name` | TEXT | VARCHAR(255) | 可空 | 导出文件原始名称 |
| `progress` | INTEGER | INT | NOT NULL, DEFAULT 0 | 导出进度百分比 0-100 |
| `error_message` | TEXT | TEXT | 可空 | 失败时的错误信息 |
| `period_start` | TEXT | VARCHAR(20) | 可空 | 报表时间段起始日期，格式 `YYYY-MM-DD` |
| `period_end` | TEXT | VARCHAR(20) | 可空 | 报表时间段结束日期，格式 `YYYY-MM-DD` |
| `created_by` | TEXT | VARCHAR(36) | NOT NULL | 创建人用户 ID（admin/formulist 均可创建导出任务） |
| `created_at` | TEXT | TIMESTAMP | NOT NULL, DEFAULT now | 创建时间 |
| `updated_at` | TEXT | TIMESTAMP | NOT NULL, DEFAULT now | 更新时间 |
| `completed_at` | TEXT | TIMESTAMP | 可空 | 完成时间 |

#### 关键设计决策

1. **`formula_id` 改为可空 + `ON DELETE SET NULL`**：原料导出和报表导出不关联配方，配方被删除后任务记录仍需保留（外键策略从 CASCADE 改为 SET NULL）
2. **移除 `api_endpoint` 字段**：API 推送不属于文件导出范畴，相关功能由 `api_data_interfaces` 表独立管理（已废弃）
3. **`target_ids_json` 设计**：当 `data_category` 为 `formula` 或 `material` 时，存储批量导出的目标 ID 列表；单条导出时可为 NULL（此时用 `formula_id`）
4. **`period_start`/`period_end`**：仅当 `data_category` 为 `weekly-report` 或 `monthly-report` 时使用
5. **`created_by` 用于数据隔离**：admin 可查看所有用户的导出任务，formulist 仅可查看自己创建的任务（通过 `created_by` 过滤）

#### `target_ids_json` 数据结构

```json
["uuid-1", "uuid-2", "uuid-3"]
```

#### 不同 `data_category` 下字段使用矩阵

| data_category | formula_id | target_ids_json | period_start | period_end |
|---------------|-----------|----------------|-------------|-----------|
| `formula` | ✅ 必填（单个）或 NULL（批量） | 批量时必填 | ❌ | ❌ |
| `material` | ❌ NULL | ✅ 必填 | ❌ | ❌ |
| `weekly-report` | ❌ NULL | ❌ NULL | ✅ 必填 | ✅ 必填 |
| `monthly-report` | ❌ NULL | ❌ NULL | ✅ 必填 | ✅ 必填 |

#### 设计说明

1. **键值对模式**：采用 EAV（Entity-Attribute-Value）模式，便于扩展新配置项而无需 ALTER TABLE
2. **`config_type` 辅助解析**：前端/后端根据类型正确解析 `config_value`（如 number 类型需 `parseInt`，boolean 需 `=== 'true'`）

4. **`updated_by` 必填**：每次配置变更必须记录操作人
5. **权限控制**：仅 admin 角色可通过 `/api/exports/config` 接口读写此表数据
6. **前端调用方**：导出配置由 ExportCenter.vue（系统管理模块下）统一管理，而非 SystemConfig.vue

## 4. MySQL 与 SQLite 差异对齐方案

### 4.1 对齐原则

| 原则 | 说明 |
|------|------|
| **SQLite 为权威来源** | 控制器代码基于 SQLite Schema 编写，MySQL 必须对齐 |
| **类型映射** | TEXT → VARCHAR(n)/TEXT/JSON，INTEGER → INT/TINYINT(1)，REAL → DECIMAL(m,n) |
| **CHECK 约束** | SQLite 支持 CHECK，MySQL 8.0+ 也支持但项目 `adaptSQL()` 会移除，由应用层校验 |
| **时间函数** | SQLite `datetime('now')` → MySQL `CURRENT_TIMESTAMP` |
| **自更新时间** | MySQL 使用 `ON UPDATE CURRENT_TIMESTAMP`，SQLite 需应用层手动更新 |

### 4.2 类型映射规则

| SQLite 类型 | MySQL 类型 | 适用场景 |
|------------|-----------|---------|
| `TEXT PRIMARY KEY` | `VARCHAR(36) PRIMARY KEY` | UUID 主键 |
| `TEXT NOT NULL` | `VARCHAR(255) NOT NULL` | 短文本（名称、编码等） |
| `TEXT NOT NULL UNIQUE` | `VARCHAR(500) NOT NULL UNIQUE` | URL、endpoint 等 |
| `TEXT` (无长度限制) | `TEXT` | 长文本（描述、错误信息） |
| `TEXT` (JSON 数据) | `JSON` | JSON 结构化数据 |
| `INTEGER` | `INT` | 整数 |
| `INTEGER` (0/1) | `TINYINT(1)` | 布尔值 |
| `REAL` | `DECIMAL(10,2)` | 金额、重量等精确小数 |

### 4.3 需对齐的表清单

| 表名 | 对齐方向 | 优先级 | 备注 |
|------|---------|--------|------|
| `api_data_interfaces` | MySQL → 对齐 SQLite | 🔴 高 | 字段完全不同，虽然已废弃但数据仍在 |
| `share_configs` | MySQL → 对齐 SQLite | 🔴 高 | 主键名不同，生产环境功能异常 |
| `export_templates` | 双向新增字段 | 🟡 中 | 新增 category、updated_at |
| `export_jobs` | 双向新增字段 + 修改约束 | 🟡 中 | 新增字段 + formula_id 可空化 |

> **注意**：`api_data_interfaces` 虽已废弃，但由于数据仍在表中，MySQL 版本仍需与 SQLite 对齐，避免数据不一致。如果未来确认不再需要此表数据，可考虑在后续版本中彻底删除。
