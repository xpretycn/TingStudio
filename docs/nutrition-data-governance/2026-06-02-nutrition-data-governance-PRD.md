# 营养数据治理 PRD

## 1. 执行摘要

### 1.1 问题陈述

TingStudio 的原料营养数据当前仅支持单源存储（`material_nutrition` 表每原料仅一条记录），所有录入方式（手动、Excel 导入、AI 解析）均直接覆盖同一份数据，导致：

1. **多源数据无法共存**：外部 API 获取的营养数据与手动录入数据互相覆盖，无法对比验证
2. **无溯源能力**：营养数据被覆盖后无法追溯原始来源，无法判断数据可信度
3. **历史不可复现**：原料营养数据变更后，已计算的配方结果自动使用新数据，历史计算结果无法复现
4. **数据质量无保障**：`data_source`、`confidence` 字段存在但计算时完全忽略，无法区分"实验室检测"和"AI 估算"

### 1.2 解决方案

引入**三层营养数据治理架构**：

- **来源层**（`material_nutrition_sources`）：多源数据独立存储，互不覆盖
- **权威层**（`material_nutrition` 增强）：管理员字段级审定"官方"数据，每个字段可溯源
- **快照层**（`formula_nutrition_snapshots`）：配方计算时锁定营养数据版本，确保历史可复现

### 1.3 成功标准

| 指标 | 目标值 | 度量方式 |
|------|--------|---------|
| 同一原料支持的最大来源数 | ≥ 5 条 | 数据库记录数 |
| 权威数据字段级溯源覆盖率 | ≥ 90% | `field_sources_json` 非空字段占比 |
| 配方计算结果可复现率 | 100% | 快照命中 vs 重新计算结果一致性 |
| 现有计算逻辑改动量 | 0 行 | `nutritionService` / `nutritionEngine` 代码 diff |
| 外部 API 数据写入来源层成功率 | ≥ 95% | 天行 API 返回结果写入 `material_nutrition_sources` 成功数/调用数 |

## 2. 用户角色与权限

### 2.1 用户角色

| 角色 | 权限 | 典型场景 |
|------|------|---------|
| **admin** | 全部权限：查看来源、对比来源、字段级选定权威、删除来源、批量补全 | 审核营养数据质量，决定采用哪个来源的数据 |
| **formulist** | 受限权限：查看来源、对比来源、添加来源（仅自己创建的原料） | 录入营养数据时查看外部 API 建议值 |

### 2.2 权限矩阵

| 操作 | admin | formulist(own) | formulist(other) |
|------|-------|---------------|-----------------|
| 查看来源列表 | ✅ | ✅ | ✅ |
| 查看来源对比 | ✅ | ✅ | ✅ |
| 添加来源数据 | ✅ | ✅ | ❌ |
| 字段级选定权威 | ✅ | ❌ | ❌ |
| 软删除来源 | ✅ | ❌ | ❌ |
| 批量补全 | ✅ | ❌ | ❌ |
| 触发外部 API 获取 | ✅ | ✅ | ❌ |

## 3. 用户故事与验收标准

### US-1：查看原料营养数据来源

**作为** admin/formulist，**我希望** 在原料详情页看到每个营养素值的数据来源，**以便** 判断数据的可信度。

**验收标准：**
- [ ] 原料详情页每个营养素值旁显示来源图标（🌐 API / 📚 种子库 / 🤖 AI / ✏️ 手动 / 📊 Excel）
- [ ] 鼠标悬停来源图标时，tooltip 显示来源详情（如"天行数据-山药"、"中国食物成分表第6版"）
- [ ] 营养数据区域顶部显示权威数据来源构成摘要（如"3项来自种子库，2项来自API"）
- [ ] 无来源信息的旧数据显示"来源未知"标签

### US-2：查看多源营养数据对比

**作为** admin/formulist，**我希望** 在原料详情页查看同一原料所有来源的营养数据对比，**以便** 了解不同来源间的数据差异。

**验收标准：**
- [ ] 原料详情页新增"查看所有来源"按钮
- [ ] 点击后展开来源对比面板，以表格形式展示所有来源数据
- [ ] 表格行 = 营养素，列 = 各来源 + 权威值
- [ ] 差异值高亮显示（与权威值不同的单元格标黄）
- [ ] 每列顶部显示来源类型图标 + 来源详情 + 可信度标签
- [ ] 无来源数据的原料显示"暂无多源数据"

### US-3：字段级选定权威数据

**作为** admin，**我希望** 从多个来源中逐字段选择最可信的值组成权威数据，**以便** 确保配方计算使用最可靠的数据。

**验收标准：**
- [ ] 来源对比面板中，每个营养素行提供下拉选择器，可切换来源
- [ ] 切换来源后，该营养素值实时更新为所选来源的值
- [ ] 选定后点击"保存权威数据"，系统将选定字段写入 `material_nutrition`
- [ ] 保存时自动更新 `field_sources_json`，记录每个字段的来源
- [ ] 保存成功后，配方计算自动使用新的权威数据
- [ ] 仅 admin 可执行此操作

### US-4：从外部 API 获取营养数据

**作为** admin/formulist，**我希望** 点击按钮从天行数据 API 获取原料营养数据，**以便** 快速填充缺失的营养数据。

**验收标准：**
- [ ] 原料详情页/编辑页新增"智能获取营养数据"按钮
- [ ] 点击后显示加载态，调用后端 API 获取天行数据
- [ ] 获取成功后，数据写入 `material_nutrition_sources`（source_type='tianapi'），不覆盖现有权威数据
- [ ] 前端自动展开来源对比面板，展示新获取的数据
- [ ] 若 API 未命中，提示"未找到匹配的营养数据"
- [ ] 若原料名称与 API 搜索词不匹配，尝试别名映射（如"薏苡仁"→"薏米"）
- [ ] API 调用失败时，提示具体错误信息，不影响现有数据

### US-5：批量补全营养数据

**作为** admin，**我希望** 一键为所有缺失营养数据的原料从外部 API 获取数据，**以便** 快速提升数据覆盖率。

**验收标准：**
- [ ] 原料列表页新增"批量补全营养数据"按钮（仅 admin 可见）
- [ ] 点击后弹出确认对话框，显示将处理的原料数量
- [ ] 执行过程中显示进度条（已处理 X/总数 Y）
- [ ] 每个原料的获取结果独立记录（成功/未命中/失败）
- [   完成后显示统计报告：成功 N 条、未命中 M 条、失败 K 条
- [ ] 所有获取的数据写入 `material_nutrition_sources`，不自动设为权威
- [ ] 支持中途取消

### US-6：配方营养计算快照

**作为** admin/formulist，**我希望** 配方计算结果锁定当时的营养数据版本，**以便** 历史结果可复现。

**验收标准：**
- [ ] 每次配方营养计算完成后，系统自动保存快照到 `formula_nutrition_snapshots`
- [ ] 快照包含：每个原料使用的营养数据 ID、版本号、来源类型、实际值
- [ ] 查看历史配方营养数据时，优先读取快照而非重新计算
- [ ] 原料营养数据变更后，已快照的配方结果不受影响
- [ ] 快照记录计算时间和计算人

### US-7：原料列表营养数据来源筛选

**作为** admin/formulist，**我希望** 在原料列表中按营养数据来源筛选原料，**以便** 快速定位需要补全或审核的原料。

**验收标准：**
- [ ] 原料列表新增筛选条件："营养数据来源"
- [ ] 选项：全部 / 仅手动录入 / 仅外部API / 仅种子库 / 无营养数据
- [ ] 列表项显示来源类型小图标
- [ ] 筛选结果实时更新

## 5. 数据来源类型定义

### 5.1 source_type 枚举

| 值 | 中文名 | 图标 | 说明 | 可信度默认值 |
|----|--------|------|------|-------------|
| `manual` | 手动录入 | ✏️ | 用户在编辑页手动填写 | medium |
| `seed` | 种子库 | 📚 | 本地预置的权威数据（来自《中国食物成分表》） | high |
| `tianapi` | 天行数据 | 🌐 | 天行数据 API 返回的结果 | medium |
| `ai` | AI 估算 | 🤖 | AI 模型生成的估算值 | low |
| `excel_import` | Excel 导入 | 📊 | 通过 Excel 文件批量导入 | medium |
| `other` | 其他 | 📎 | 其他来源 | medium |

### 5.2 来源优先级（自动推荐时使用，非强制）

```
seed > tianapi > excel_import > manual > ai > other
```

## 6. 业务规则

### 6.1 权威数据选定规则

1. 权威数据由 admin 人工指定，系统不自动切换
2. 字段级选定：每个营养素可独立选择来源
3. 选定后 `material_nutrition.per_100g_json` 更新为选定值
4. `material_nutrition.field_sources_json` 记录每个字段的来源溯源
5. `material_nutrition.source_type` 设为 `composite`（当多来源组合时）或单一来源类型

### 6.2 来源数据管理规则

1. 同一原料同一来源类型仅保留一条活跃记录（`is_active = 1`）
2. 新增同类型来源时，旧记录自动软删除（`is_active = 0`）
3. 软删除的来源数据不参与对比，但保留历史记录
4. 来源数据不可编辑，只能新增或软删除

### 6.3 快照规则

1. 每次配方营养计算完成后自动创建快照
2. 同一配方的同一版本仅保留最新快照（覆盖旧快照）
3. 快照数据不可修改
4. 快照中的 `per100gSnapshot` 保存当时使用的实际营养值

### 6.4 外部 API 调用规则

1. 天行数据 API 调用需要 `TIANAPI_KEY` 环境变量
2. 功能开关 `EXTERNAL_NUTRITION_ENABLED` 控制是否启用
3. API 返回数据写入来源层，不直接写入权威层
4. 能量值自动换算：kcal × 4.184 = kJ
5. API 调用失败时记录日志，不影响现有数据

## 7. 风险分析

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|---------|
| 迁移脚本将现有数据错误分类 | 中 | 低 | 迁移后提供管理界面校正；所有旧数据默认归为 `manual` |
| 天行数据 API 停机 | 中 | 中 | 来源层独立存储，API 不可用不影响现有数据；本地种子库作为 fallback |
| 管理员选定错误权威数据 | 低 | 高 | 字段级溯源可追溯；支持重新选定 |
| 快照表数据膨胀 | 低 | 低 | 每配方每版本仅保留最新快照；定期归档 |
| `field_sources_json` 查询性能 | 低 | 低 | 该字段仅展示用，不参与计算，按需查询 |
| 前端来源对比面板渲染慢 | 低 | 低 | 来源数量有限（通常 ≤ 5），表格行数固定（28 种营养素） |

## 9. 验收测试场景

### 场景 1：多源数据共存

1. 原料"山药"已有手动录入的营养数据
2. 点击"智能获取"，天行 API 返回山药营养数据
3. 验证：`material_nutrition_sources` 新增一条 source_type='tianapi' 的记录
4. 验证：`material_nutrition` 的权威数据未被修改
5. 验证：来源对比面板同时展示两条来源数据

### 场景 2：字段级权威选定

1. 原料"山药"有 2 条来源数据（手动 + 天行 API）
2. admin 在对比面板中将"蛋白质"从手动来源切换为天行 API 来源
3. 点击"保存权威数据"
4. 验证：`material_nutrition.per_100g_json` 中蛋白质值更新为天行 API 的值
5. 验证：`material_nutrition.field_sources_json` 中蛋白质字段记录来源为 tianapi
6. 验证：`material_nutrition.source_type` 更新为 `composite`

### 场景 3：配方快照可复现

1. 配方 A 使用原料"山药"计算营养数据，结果保存快照
2. 原料"山药"的权威营养数据被更新
3. 重新查看配方 A 的营养数据
4. 验证：显示快照中的旧数据，而非重新计算的新数据

### 场景 4：向后兼容

1. 现有 API `PUT /api/nutrition/material/:id` 仍可正常保存营养数据
2. 保存时自动将数据同步写入 `material_nutrition_sources`（source_type='manual'）
3. 现有配方计算逻辑无需修改，仍从 `material_nutrition` 读取
