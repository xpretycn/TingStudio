# 权限管理功能 — 产品需求文档（PRD）

## 1. Executive Summary

### Problem Statement

TingStudio 当前权限体系为硬编码二元模型（admin 拥有一切，formulist 受限），无法灵活配置角色权限。新增角色或调整权限需要修改代码并重新部署，无法满足业务扩展需求。

### Proposed Solution

在系统工具模块中新增「权限管理」功能，实现 RBAC（基于角色的访问控制），支持自定义角色、模块级权限分配、用户角色管理，让管理员可通过界面灵活配置权限。

### Success Criteria

1. 管理员可在界面上创建/编辑/删除自定义角色，并分配模块级权限
2. 管理员可在界面上查看用户列表、分配角色、禁用/启用用户
3. 现有 admin 角色保持超级权限，不受权限配置影响
4. 现有 formulist 角色自动迁移为默认权限配置，功能不降级
5. 权限变更后，用户下次请求自动生效（无需重新登录）

## 2. User Experience & Functionality

### 2.1 User Personas

| 角色 | 描述 | 核心诉求 |
|------|------|---------|
| **管理员（admin）** | 系统管理者，拥有全部权限 | 灵活管理角色和权限，控制用户访问范围 |
| **配方师（formulist）** | 普通用户，权限受限 | 正常使用被授权的功能模块 |
| **自定义角色用户** | 新增角色的使用者 | 仅可访问被授权的模块 |

### 2.2 User Stories

#### US-1：角色管理

**As a** 管理员，**I want to** 在系统工具中查看和管理所有角色，**so that** 我可以灵活定义不同用户的权限范围。

**Acceptance Criteria：**
- 可查看所有角色列表（名称、描述、权限数量、关联用户数）
- 系统内置角色（admin、formulist）标记为「系统角色」，不可删除，admin 的权限不可编辑
- 可创建新角色（输入名称、描述）
- 可编辑角色（修改名称、描述、权限配置）
- 可删除自定义角色（需先解除该角色下的用户关联）
- 角色名称不可重复

#### US-2：权限分配

**As a** 管理员，**I want to** 为角色分配模块级权限，**so that** 不同角色用户只能访问被授权的功能。

**Acceptance Criteria：**
- 权限以模块为单位展示，每个模块有「读」和「写」两个权限
- 可通过勾选方式批量分配权限
- admin 角色始终拥有全部权限，权限配置区域显示为只读
- 权限变更实时保存
- 支持一键全选/取消全选

#### US-3：用户管理

**As a** 管理员，**I want to** 查看用户列表并管理其角色和状态，**so that** 我可以控制每个用户的系统访问能力。

**Acceptance Criteria：**
- 可查看所有用户列表（用户名、显示名、角色、状态、创建时间）
- 可按用户名/角色/状态筛选
- 可修改用户角色（下拉选择已有角色）
- 可禁用/启用用户（禁用后无法登录）
- 不可禁用自己
- 不可修改自己的角色
- 用户角色变更后，该用户下次请求自动生效

#### US-4：权限管理 Tab 入口

**As a** 管理员，**I want to** 在系统工具页面中快速进入权限管理，**so that** 我无需额外导航即可管理权限。

**Acceptance Criteria：**
- SystemConfig 页面新增「权限管理」Tab
- 仅 admin 角色可见该 Tab
- Tab 内包含「角色管理」和「用户管理」两个子 Tab

### 2.3 Non-Goals

- 本次**不**重构现有控制器中的硬编码权限判断（`role !== "admin"`），保持 admin 超级权限的现有逻辑不变
- 本次**不**实现前端路由级别的权限守卫（`requiresAdmin` meta 已定义但未生效，后续迭代处理）
- 本次**不**实现操作级细粒度权限（如"原料删除"单独授权），仅支持模块级
- 本次**不**实现权限变更审计日志

## 2.4 详细交互界面设计

### 2.4.1 整体布局

权限管理集成在 SystemConfig 页面中，作为新增的顶级 Tab，与「缓存配置」「含量比配置」「数据概览」「原料值管理」「导出中心」并列。

**Tab 栏变更**：
- 新增第 6 个 Tab：「权限管理」，仅 admin 可见
- 副标题更新为：「缓存配置、含量比校验、原料值管理、权限管理与系统状态」

**权限管理 Tab 内部布局**：采用与 EnumManage / ExportCenter 一致的「左侧导航 + 右侧内容」双栏布局：
- 左侧导航栏（可折叠）：角色管理 / 用户管理
- 右侧内容区：根据左侧选中项展示对应面板

## 3. 权限模块定义

### 3.1 模块级权限清单

| 模块 | 读权限标识 | 写权限标识 | 说明 |
|------|-----------|-----------|------|
| 原料管理 | `material:read` | `material:write` | 原料 CRUD、审批、版本管理 |
| 配方管理 | `formula:read` | `formula:write` | 配方 CRUD、审批、发布、版本 |
| AI 助手 | `ai:read` | `ai:write` | AI 对话、NL2SQL、智能推荐 |
| 营养分析 | `nutrition:read` | `nutrition:write` | 营养成分计算、营养报告 |
| 文件管理 | `file:read` | `file:write` | 文件上传/删除/管理 |
| 数据导出 | `export:read` | `export:write` | 导出 Excel/PDF、导出中心 |
| 系统配置 | `system:read` | `system:write` | 缓存/含量比/枚举值/模型管理 |
| 用户管理 | `user:read` | `user:write` | 用户列表/角色分配/禁用启用 |
| 权限管理 | `permission:read` | `permission:write` | 角色 CRUD/权限配置 |

### 3.2 默认角色权限配置

| 模块 | admin | formulist |
|------|-------|-----------|
| 原料管理 | 全部 | 读 + 写（仅自己创建的） |
| 配方管理 | 全部 | 读 + 写（仅自己创建的） |
| AI 助手 | 全部 | 读 + 写 |
| 营养分析 | 全部 | 读 + 写 |
| 文件管理 | 全部 | 读 |
| 数据导出 | 全部 | 读 + 写 |
| 系统配置 | 全部 | 无 |
| 用户管理 | 全部 | 无 |
| 权限管理 | 全部 | 无 |

> **注意**：formulist 的数据隔离逻辑（仅见自己创建的数据）由控制器层 `WHERE created_by = ?` 实现，不在权限层控制。权限层仅控制功能模块的访问权。

## 4. Technical Specifications

### 4.1 Architecture Overview

```
前端 SystemConfig.vue
  └─ 新增 Tab: 权限管理
       ├─ 子 Tab: 角色管理 → RoleManage.vue
       │    ├─ 角色列表（t-table）
       │    ├─ 新增/编辑角色弹窗（t-dialog + t-form）
       │    └─ 权限分配面板（t-checkbox-group）
       └─ 子 Tab: 用户管理 → UserManage.vue
            ├─ 用户列表（t-table + 筛选）
            ├─ 角色分配（t-select）
            └─ 禁用/启用（t-switch / t-popconfirm）

后端 API
  ├─ /api/roles          — 角色 CRUD
  ├─ /api/roles/:id/permissions — 角色权限分配
  ├─ /api/permissions    — 权限列表查询
  └─ /api/users          — 用户管理（列表/角色分配/禁用启用）

数据库
  ├─ roles              — 角色表
  ├─ permissions        — 权限表（系统预置）
  ├─ role_permissions   — 角色-权限关联表
  └─ users.role_id      — 用户表新增角色关联字段
```

### 4.2 Integration Points

- **认证中间件**：`generateToken()` 从数据库读取角色权限写入 JWT，`requirePermission()` 保持现有逻辑
- **Auth Store**：前端 `useAuthStore` 扩展 `permissions` 字段，提供 `hasPermission()` 方法
- **路由注册**：新增 `/api/roles`、`/api/permissions`、扩展 `/api/users` 路由

### 4.3 Security & Privacy

- 权限管理 API 全部需要 admin 角色
- 用户管理 API 需要 `user:read` / `user:write` 权限
- 禁用用户后，该用户的 JWT 在过期前仍可使用（7 天有效期），如需即时生效需额外实现 Token 黑名单（本次不实现）
- 不可禁用/修改 admin 角色本身

### 5.2 Technical Risks

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 现有硬编码权限判断与新权限体系并存 | 部分功能可能绕过新权限体系 | admin 保持超级权限，新权限体系仅影响非 admin 角色 |
| JWT 7 天有效期导致权限变更不即时生效 | 用户角色变更后最长 7 天才生效 | 本次接受此限制，v1.1 考虑 Token 刷新机制 |
| 数据迁移风险（现有用户角色映射） | 迁移错误可能导致用户权限丢失 | 迁移脚本幂等，admin 角色不受影响 |
