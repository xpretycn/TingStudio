# PRD：配方营养分析功能重构

## 1. 执行摘要

### 1.1 问题陈述

当前 TingStudio 的营养分析页面（NutritionAnalysis.vue）偏展示型仪表盘，缺乏实际分析指导价值：用户选择配方后只能看到数值罗列，无法解读"好还是不好"、"该怎么调"，且计算逻辑存在两套并行实现（控制器内联 + NutritionEngine 类），合规检查与食品行业实际标准脱节。

### 1.2 解决方案

重构营养分析功能，对标 **GB 28050-2011（预包装食品营养标签通则）** 和 **GB 14880-2012（食品营养强化剂使用标准）**，将营养分析从"数据展示"升级为"分析决策工具"：自动生成国标营养标签、原料贡献分析、数据覆盖度检查、营养参考对比与含量声称判定。

### 1.3 成功标准

| KPI | 指标 |
|-----|------|
| 计算引擎统一 | 消除两套并行实现，NutritionEngine 为唯一计算入口 |
| 国标合规 | 营养标签格式 100% 符合 GB 28050 要求 |
| 数据覆盖度 | 分析结果必须展示原料营养数据覆盖率，缺失数据明确标注 |
| 结果可解读 | 每项营养素都有 NRV% 对比 + 含量声称判定 + 文字解读 |
| 操作效率 | 从选择配方到获得完整分析结果 ≤ 3 次点击 |

## 2. 用户体验与功能

### 2.1 用户画像

| 角色 | 需求 |
|------|------|
| **配方师（formulist）** | 快速了解自己配方的营养构成，判断是否符合国标，获取调整方向 |
| **管理员（admin）** | 审核所有配方的营养合规性，对比不同配方的营养差异 |

### 2.2 用户故事

#### US-1：营养标签生成
> 作为配方师，我想查看配方的国标营养标签，以便了解成品每100g的营养含量和NRV占比。

**验收标准**：
- 选择配方后，自动生成 GB 28050 格式的营养成分表（方框表）
- 强制标示项：能量、蛋白质、脂肪、碳水化合物、钠（1+4）
- 扩展标示项：膳食纤维、糖、钙、铁、锌、维生素A、维生素C、维生素D、维生素E、维生素B1、维生素B2、维生素B6、维生素B12、叶酸、钾
- 每项显示：含量值 + 单位 + NRV%
- 0界限归零：能量≤17kJ、蛋白质≤0.5g、脂肪≤0.5g、碳水≤0.5g、钠≤5mg → 标示为"0"
- 归零后重新计算能量

#### US-2：原料贡献分析
> 作为配方师，我想了解每种原料对成品营养的贡献占比，以便识别关键营养来源和调整方向。

**验收标准**：
- 展示每种原料的：名称、类型（药材/辅料）、用量、含量比（ratio）
- 展示每种原料对核心营养素的贡献值和贡献占比
- 区分药材（ratio_factor 默认 0.18）和辅料（ratio_factor 默认 1.0）
- 支持展开查看某原料的完整营养素明细

#### US-3：数据覆盖度检查
> 作为配方师，我想知道哪些原料缺少营养数据，以便评估分析结果的可信度。

**验收标准**：
- 统计配方中原料的营养数据覆盖率（有数据的原料数 / 总原料数）
- 明确标注哪些原料缺少营养数据
- 缺失数据的原料在贡献分析中标记为"数据缺失"
- 覆盖率低于 80% 时，在分析结果顶部显示警告提示

#### US-4：营养参考对比与含量声称
> 作为配方师，我想知道配方营养是否符合国标参考值，以及可以做出哪些营养声称。

**验收标准**：
- 将每项营养素与 GB 28050 NRV 参考值对比，标注：偏低（<10% NRV）、适中（10%-30% NRV）、偏高（>30% NRV）
- 基于 GB 28050 附录C.1 含量声称条件，自动判定可用的营养声称：
  - "低脂肪"：≤3g/100g（固体）
  - "低钠"：≤120mg/100g
  - "低糖"：≤5g/100g（固体）
  - "高蛋白质"：≥20% NRV/100g（即≥12g/100g）
  - "蛋白质来源"：≥10% NRV/100g（即≥6g/100g）
  - "高钙"：≥30% NRV/100g（即≥240mg/100g）
  - "钙来源"：≥15% NRV/100g（即≥120mg/100g）
  - "膳食纤维来源"：≥10% NRV/100g（即≥2.5g/100g）
  - "高膳食纤维"：≥20% NRV/100g（即≥5g/100g）
  - "无反式脂肪"：≤0.3g/100g
- 不可用声称也列出（当前值不满足条件），并显示差距值

#### US-5：营养强化剂合规检查（GB 14880）
> 作为配方师，我想知道配方中的辅料（营养强化剂）使用量是否符合 GB 14880 规定。

**验收标准**：
- 识别配方中标记为 supplement 类型的原料
- 对照 GB 14880 附录A，检查各强化剂的使用量是否在允许范围内
- 标注：合规（在范围内）、偏低（低于下限）、偏高（高于上限）
- 对于无对应标准的强化剂，标注"未在标准中找到对应规定"

#### US-6：分析报告摘要
> 作为配方师，我想看到一份简洁的分析摘要，快速了解配方的营养状况。

**验收标准**：
- 顶部展示关键指标卡片：数据覆盖率、核心营养素达标率、可用营养声称数、合规检查结果
- 每个指标卡片有颜色标识：绿色（良好）、黄色（注意）、红色（异常）
- 一句话总结：如"该配方蛋白质含量丰富，可声称'高蛋白质'；钠含量偏高，建议关注"

### 2.3 非目标（不在本次范围）

- 多配方营养对比功能
- AI 驱动的配方优化建议
- 营养标签导出为图片/PDF
- 历史分析记录保存与回溯
- 营养标准管理页面重构（保留现有功能）

## 3. 技术规格

### 3.1 架构概览

```
前端 NutritionAnalysis.vue
  ├─ 选择配方 → 调用 API
  ├─ 营养标签组件（GB 28050 方框表）
  ├─ 原料贡献组件
  ├─ 数据覆盖度组件
  ├─ 含量声称组件
  └─ 强化剂合规组件

后端 API
  ├─ POST /api/nutrition/analyze/:formulaId  — 一键分析（合并计算+对比+声称+合规）
  ├─ GET  /api/nutrition/coverage/:formulaId — 数据覆盖度检查
  └─ 现有接口保持兼容

NutritionEngine（统一计算引擎）
  ├─ prepareData()         — 数据准备
  ├─ calculateRatios()     — 含量比计算
  ├─ aggregateNutrition()  — 营养贡献汇总
  ├─ calculateAtwater()    — Atwater 能量计算
  ├─ calculateNRV()        — NRV% 计算
  ├─ applyZeroThreshold()  — 0界限归零
  ├─ recalculateEnergy()   — 归零后重算能量
  ├─ evaluateClaims()      — 含量声称判定（新增）
  └─ checkFortification()  — 强化剂合规检查（新增）
```

### 3.2 计算流程（7+2 步）

**原有 7 步**（保持不变）：
1. 数据准备：统计药材/辅料数量
2. 含量比计算：`ratio = (quantity / finishedWeight) × ratioFactor`
3. 营养贡献汇总：`total += per100g[field] × ratio`
4. Atwater 能量计算：`energy = protein×17 + fat×37 + carbohydrate×17`
5. NRV% 计算：`(value / NRV_reference) × 100`
6. 0界限归零：低于阈值的归零
7. 归零后重算能量

**新增 2 步**：
8. 含量声称判定：基于 GB 28050 附录C.1 条件逐项判定
9. 强化剂合规检查：基于 GB 14880 附录A 使用量范围逐项检查

### 3.6 集成点

| 集成 | 说明 |
|------|------|
| 配方数据 | 通过 `formulas` 表 + `materials_json` 获取原料组成 |
| 原料营养 | 通过 `material_nutrition` 表获取每100g营养数据 |
| 营养标准 | 内置 GB 28050 NRV + GB 14880 强化剂标准，不依赖 `nutrition_profiles` 表 |
| 认证 | 复用现有 `authMiddleware`，admin 全量可见，formulist 仅自己创建的配方 |

### 3.7 安全与隐私

- 营养分析结果仅对已认证用户可见
- 配方数据隔离：formulist 仅能分析自己创建的配方
- 计算结果不缓存敏感信息，仅存储汇总数据

## 4. 风险与路线图

### 4.1 分阶段交付

| 阶段 | 内容 | 优先级 |
|------|------|--------|
| **MVP** | 统一计算引擎 + 营养标签 + 原料贡献 + 数据覆盖度 | P0 |
| **v1.1** | 含量声称判定 + 营养参考对比 | P1 |
| **v1.2** | 强化剂合规检查（GB 14880） | P1 |
| **v2.0** | 多配方对比 + AI 建议 + 导出 | P2 |

### 4.2 技术风险

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 原料营养数据稀疏 | 分析结果可信度低 | 数据覆盖度检查 + 缺失提示 |
| 药材 ratio_factor 差异大 | 计算结果偏差 | 明确标注 ratio_factor 取值及依据 |
| GB 14880 食品分类映射复杂 | 强化剂合规检查不完整 | 先支持常见分类，后续迭代 |
| 两套计算逻辑迁移风险 | 结果不一致 | 迁移后对比测试，确保结果一致 |

## 5. 评估策略

### 5.1 功能验证

- 选取 3 个已知配方，手动计算营养含量，与系统结果对比，误差 ≤ 5%
- 含量声称判定结果与 GB 28050 附录C.1 逐条核对
- 0界限归零逻辑与 GB 28050 表1 逐条核对

### 5.2 性能验证

- 单配方分析响应时间 ≤ 2 秒
- 10 个原料的配方分析响应时间 ≤ 3 秒
