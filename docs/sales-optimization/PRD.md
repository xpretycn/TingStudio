# PRD：销量分析模块优化

## 1. Executive Summary

**Problem Statement**：当前销量模块存在唯一约束不含业务员、后端路由未走 Controller 规范层、重复录入无合并策略、录入入口单一等问题，导致不同业务员无法分别录入同一配方同月销量，重复录入直接报错，数据管理体验差。同时，现有录入方式仅支持逐条手动录入，月末集中录入多个配方销量时效率极低。

**Proposed Solution**：调整数据库唯一约束支持业务员维度、切换后端到 Controller 标准实现、新增重复录入智能合并策略、增加业务员维度录入入口、新增在线表格批量录入功能（含复制上期数据快速填充）。

**Success Criteria**：

1. 不同业务员可分别录入同一配方同月的销量，互不冲突
2. 同一配方+同一业务员+同一周期重复录入时，提供「累加」或「覆盖」选项，不再直接报错
3. 后端路由切换到 Controller 层，created\_by 取实际用户、period\_end 按周期类型正确计算
4. 销量分析页面、配方详情页面、业务员管理页面均可作为销量录入入口
5. 月份选择器禁止选择未来月份（维持现有行为）
6. 支持在线表格批量录入：选择周期和业务员后，在可编辑表格中一次性填写多个配方的销量和金额
7. 支持复制上期数据：一键复制上月销量记录作为当月起点，仅需修改变化项
8. 批量录入支持重复记录的统一合并策略（全局累加 / 全局覆盖）

## 2. User Experience & Functionality

### 2.1 User Personas

| 角色        | 说明                          |
| --------- | --------------------------- |
| admin     | 管理员，可查看全部数据，可录入/编辑/删除任意销量记录 |
| formulist | 配方师，仅可操作自己创建的数据             |

### 2.2 User Stories

**US-1：业务员维度销量录入**

> As a admin，I want 从业务员管理页面直接录入某业务员的销量数据，so that 不必先找到配方再录入。

* AC1：业务员列表页每行操作栏新增「录入销量」按钮

* AC2：点击后打开 SalesRecordDrawer，自动选中该业务员关联的配方列表

* AC3：录入成功后刷新业务员列表

**US-2：同一配方不同业务员同月销量**

> As a admin，I want 不同业务员可以分别录入同一配方同月的销量，so that 各业务员的销售贡献独立统计。

* AC1：数据库唯一约束包含 salesman\_id

* AC2：不同业务员录入同一配方同月销量时互不冲突

* AC3：统计排行按 salesman\_id 独立计算

**US-3：重复录入智能合并**

> As a admin，I want 同一配方+同一业务员+同一周期重复录入时，可以选择累加或覆盖，so that 不会因重复录入而报错。

* AC1：录入时检测到已有记录，弹出确认对话框

* AC2：对话框显示已有记录的销量和金额，提供「累加」和「覆盖」两个选项

* AC3：选择「累加」：新销量叠加到已有记录（quantity += 新值，revenue += 新值）

* AC4：选择「覆盖」：用新值替换已有记录

* AC5：后端 API 支持 mergeMode 参数（accumulate / replace）

**US-4：后端路由规范化**

> As a developer，I want 销量路由切换到 Controller 层标准实现，so that 代码规范统一、created\_by 正确记录、period\_end 正确计算。

* AC1：路由注册切换到 salesController

* AC2：created\_by 取 req.user.userId 而非硬编码 "system"

* AC3：period\_end 根据 periodType 正确计算月末/季末/年末

* AC4：返回数据使用 rowsToCamelCase 转换

**US-5：月份选择限制**

> As a admin，I want 录入销量时不能选择未来月份，so that 避免录入无效的预测数据。

* AC1：月份选择器 disable-date 禁止未来月份（维持现有行为）

* AC2：后端也校验 periodStart 不得晚于当前月份

**US-6：在线表格批量录入**

> As a admin，I want 在一个可编辑的表格中同时填写多个配方的销量数据，so that 月末集中录入时不必逐条操作，大幅提升录入效率。

* AC1：销量分析页面新增「批量录入」按钮，点击打开 SalesBatchDrawer 抽屉组件

* AC2：抽屉顶部选择「统计周期」（月份/季度/年度）和「周期值」（如 2026-05），周期选择器禁止未来月份

* AC3：抽屉顶部可选「业务员」筛选，选择后表格仅显示该业务员关联的配方；不选则显示全部配方

* AC4：表格列为：配方名称（只读）、配方编码（只读）、业务员（下拉选择）、销量（件，可编辑）、金额（万元，可编辑）、备注（可编辑）

* AC5：表格支持 TDesign 的可编辑行模式，空行自动跳过不提交

* AC6：底部提供「全局合并策略」选择：累加 / 覆盖（默认累加）

* AC7：点击「提交」后，前端逐条校验，校验通过后调用批量 API 一次性提交

* AC8：提交结果展示成功条数和失败条数，失败行高亮显示错误原因，支持修正后重新提交失败行

* AC9：formulist 角色仅可看到自己创建的配方，admin 可见全部

**US-7：复制上期数据快速填充**

> As a admin，I want 批量录入时能一键复制上月销量数据作为起点，so that 当月销量与上月变化不大时只需修改少量数字，不必逐行重新填写。

* AC1：SalesBatchDrawer 顶部提供「复制上期」按钮，点击后弹出确认提示

* AC2：确认后，系统查询所选周期的上一周期（月度→上月、季度→上季、年度→上年）已有销量记录

* AC3：将上期记录的 quantity 和 revenue 自动填入表格对应行的可编辑列

* AC4：若某配方在上期无记录，该行销量和金额保持空白

* AC5：填充的数据为可编辑状态，用户可自由修改后再提交

* AC6：若当前周期已有部分记录，复制上期数据时对已有记录行按 mergeMode 处理（默认累加）

* AC7：复制操作不影响表格中用户已手动填写的数据（已填写行不覆盖）

### 2.3 Non-Goals

* 不做销量预测/趋势分析功能

* 不做销量审批流程

* 不做 Excel/CSV 文件导入（在线表格批量录入 + 复制上期已足够覆盖批量场景）

* 不修改销量分析页面的展示逻辑和图表

* 不做自动定时导入（如定时从外部系统拉取）

* 不做图片识别录入（OCR 识别表格图片）

### 3.2 Integration Points

* **认证**：所有销量 API 需要 authMiddleware，created\_by 从 req.user 获取

* **数据库**：formula\_sales 表唯一约束变更，需迁移脚本

* **前端组件**：SalesRecordDrawer 增加 mergeMode 逻辑；新增 SalesBatchDrawer

* **批量事务**：批量 API 逐条提交，单条失败不影响其他记录

### 3.3 Security & Privacy

* 销量数据遵循现有数据隔离规则（admin 全部可见，formulist 仅见自己创建的）

* 重复录入合并需前端确认，防止误操作

* 批量 API 限流：单次最多 200 条，防止恶意大量写入

* formulist 角色批量录入时，自动过滤无权限的配方数据行

## 4. 批量录入功能详细设计

### 4.1 业务需求分析

#### 4.1.1 现状痛点

| 痛点编号 | 描述                                           | 影响               |
| ---- | -------------------------------------------- | ---------------- |
| P1   | 月末需逐条录入 10-50 条销量数据，每条需选择配方、业务员、月份、填写数值，操作繁琐 | 录入效率低，耗时 30 分钟以上 |
| P2   | 多个业务员的月度销量需分别录入，无法一次性完成                      | 操作步骤多，容易遗漏       |
| P3   | 当月销量与上月变化不大时，仍需逐条重新填写所有数值                    | 重复劳动，容易出错        |

#### 4.1.2 目标场景

| 场景编号 | 场景描述                          | 推荐录入方式          | 预估数据量   |
| ---- | ----------------------------- | --------------- | ------- |
| S1   | 月末集中录入：管理员汇总各业务员本月销量后一次性录入    | 在线表格批量录入        | 10-50 条 |
| S2   | 单业务员多配方：录入某业务员负责的所有配方月度销量     | 在线表格批量录入（筛选业务员） | 5-20 条  |
| S3   | 稳定销量快速续录：当月销量与上月基本一致，仅个别配方有变化 | 复制上期 + 修改       | 10-30 条 |

### 4.2 功能模块设计

#### 4.2.1 SalesBatchDrawer（在线表格批量录入 + 复制上期）

**组件定位**：抽屉式组件，复用现有 SalesRecordDrawer 的样式风格，内部使用 TDesign 可编辑表格。集成「复制上期」快捷操作。

**组件 Props**：

| Prop               | 类型      | 必填 | 说明                  |
| ------------------ | ------- | -- | ------------------- |
| visible            | boolean | 是  | 控制抽屉显隐              |
| defaultSalesmanId  | string  | 否  | 预选业务员 ID            |
| defaultPeriodType  | string  | 否  | 预选周期类型，默认 'monthly' |
| defaultPeriodStart | string  | 否  | 预选周期值，如 '2026-05'   |

**组件 Emits**：

| Event          | Payload     | 说明     |
| -------------- | ----------- | ------ |
| update:visible | boolean     | 抽屉关闭   |
| success        | BatchResult | 批量录入成功 |

**组件内部结构**：详见 4.4.2 节「SalesBatchDrawer 抽屉界面设计」。

**表格行为规则**：

1. 选择周期后，表格自动加载配方列表（含配方名称、编码、已关联业务员）
2. 业务员列默认填充配方关联的业务员，可下拉切换（下拉数据来自 salesmen 表活跃状态记录）
3. 销量和金额列为空则该行不提交；销量有值但金额为空时，金额默认为 0
4. 表格支持滚动，默认展示 20 行，超过时出现滚动条
5. 表格底部显示统计：已填写 N 行 / 共 M 行

**「复制上期」交互流程**：详见 4.4.4 节「复制上期数据交互流程」。

### 4.3 数据格式规范

#### 4.3.1 批量 API 请求格式

**POST /api/sales/batch**

请求体：

```json
{
  "records": [
    {
      "formulaId": "uuid-001",
      "salesmanId": "uuid-002",
      "periodType": "monthly",
      "periodStart": "2026-05",
      "quantity": 100,
      "revenue": 5.2,
      "notes": ""
    }
  ],
  "mergeMode": "accumulate"
}
```

| 字段                     | 类型     | 必填 | 说明                             |
| ---------------------- | ------ | -- | ------------------------------ |
| records                | array  | 是  | 销量记录数组，最多 200 条                |
| records\[].formulaId   | string | 是  | 配方 UUID                        |
| records\[].salesmanId  | string | 是  | 业务员 UUID                       |
| records\[].periodType  | string | 否  | 周期类型，默认 monthly                |
| records\[].periodStart | string | 是  | 周期起始，格式 YYYY-MM                |
| records\[].quantity    | number | 是  | 销量，≥ 0 整数                      |
| records\[].revenue     | number | 是  | 金额，≥ 0                         |
| records\[].notes       | string | 否  | 备注                             |
| mergeMode              | string | 否  | 全局合并策略：accumulate（默认）/ replace |

#### 4.3.2 批量 API 响应格式

**成功响应**：

```json
{
  "success": true,
  "data": {
    "total": 10,
    "succeeded": 8,
    "failed": 1,
    "skipped": 1,
    "results": [
      {
        "index": 0,
        "status": "success",
        "recordId": "uuid-xxx",
        "formulaCode": "FP-001",
        "salesmanName": "张三",
        "action": "created",
        "message": "新建成功"
      },
      {
        "index": 1,
        "status": "merged",
        "recordId": "uuid-yyy",
        "formulaCode": "FP-002",
        "salesmanName": "李四",
        "action": "accumulated",
        "message": "已累加到已有记录（原销量: 100 → 300，原金额: 5.0 → 13.5）"
      },
      {
        "index": 2,
        "status": "skipped",
        "formulaCode": "FP-003",
        "salesmanName": "张三",
        "action": "skipped",
        "message": "销量和金额均为空，跳过"
      },
      {
        "index": 3,
        "status": "failed",
        "formulaCode": "FP-999",
        "salesmanName": "张三",
        "action": "none",
        "message": "配方不存在"
      }
    ]
  }
}
```

| 字段                 | 说明                                                |
| ------------------ | ------------------------------------------------- |
| total              | 总处理条数                                             |
| succeeded          | 成功条数（含新建和合并）                                      |
| failed             | 失败条数                                              |
| skipped            | 跳过条数（空行等）                                         |
| results            | 逐条处理结果                                            |
| results\[].index   | 原始数据行索引（0-based）                                  |
| results\[].status  | success / merged / skipped / failed               |
| results\[].action  | created / accumulated / replaced / skipped / none |
| results\[].message | 处理结果描述                                            |

**新增按钮**：在现有 `add-formula-btn`（「录入销量」）按钮后面，新增一个 `add-formula-btn` 样式按钮「批量录入」。

* 按钮文案：「批量录入」

**位置**：工具栏右侧按钮区域，现有「创建新配方」按钮左侧。

* 按钮文案：「批量录入销量」

##### 入口三：业务员管理页（SalesmanList.vue）— 仅单条录入

**位置**：操作列现有「录入销量」按钮（`sales-btn` class），不新增批量入口。

**原因**：业务员管理页的核心场景是「为某个业务员录入销量」，属于单条录入场景。批量录入的维度是「某个周期多个配方」，而非「某个业务员多个配方」，因此业务员页面不适合作为批量录入入口。

**保持不变**：

* 操作列 `sales-btn` 按钮 → 打开 SalesRecordDrawer（单条），传入 `salesmanId`

* 不新增批量录入按钮

#### 4.4.4 复制上期数据交互流程

```

  - 年度 → 上年（2026 → 2025）
        │
        ▼

  「将从 {上期周期值} 复制销量数据到当前表格，已手动填写的数据不会被覆盖，是否继续？」
        │
        ├── 取消 ──▶ 无操作
        │
        └─ 确认
            │

            │

            │
            ▼
```

| 场景          | 处理方式              |
| ----------- | ----------------- |
| 上期无任何销量记录   | 提示「上期无销量数据可复制」    |

### 4.5 数据校验规则

#### 4.5.1 前端校验（提交前）

| 校验编号  | 校验项          | 校验规则                                                         | 错误级别 | 错误提示              |
| ----- | ------------ | ------------------------------------------------------------ | ---- | ----------------- |
| V-F01 | 业务员必选        | salesmanId 不为空                                               | 错误   | 第 N 行：请选择业务员      |
| V-F02 | 销量格式         | quantity 为非负整数                                               | 错误   | 第 N 行：销量必须为非负整数   |
| V-F03 | 金额格式         | revenue 为非负数                                                 | 错误   | 第 N 行：金额必须为非负数    |
| V-F04 | 重复行检测        | 同一表格内 formulaId + salesmanId + periodType + periodStart 组合唯一 | 警告   | 第 N 行与第 M 行数据重复   |
| V-F05 | formulist 权限 | formulist 角色仅可录入自己创建的配方                                      | 错误   | 第 N 行：无权限录入该配方的销量 |

#### 4.5.2 后端校验（处理时）

| 校验编号  | 校验项             | 校验规则                                              | 错误级别           |
| ----- | --------------- | ------------------------------------------------- | -------------- |
| V-B01 | 请求体 records 数组  | 非空数组，长度 ≤ 200                                     | 错误（400）        |
| V-B02 | 单条记录必填字段        | formulaId、salesmanId、periodStart、quantity、revenue | 错误（逐条标记）       |
| V-B03 | formulaId 外键    | formulas 表中存在                                     | 错误（逐条标记）       |
| V-B04 | salesmanId 外键   | salesmen 表中存在                                     | 错误（逐条标记）       |
| V-B05 | periodStart 非未来 | ≤ 当前月份                                            | 错误（逐条标记）       |
| V-B06 | quantity 非负整数   | ≥ 0                                               | 错误（逐条标记）       |
| V-B07 | revenue 非负数     | ≥ 0                                               | 错误（逐条标记）       |
| V-B08 | formulist 数据隔离  | formulist 仅可操作 created\_by = req.user.userId 的配方  | 错误（逐条标记）       |
| V-B09 | 重复记录处理          | 根据 mergeMode 处理：accumulate 累加 / replace 覆盖        | 正常（标记为 merged） |
| V-B10 | periodEnd 计算    | 根据 periodType 自动计算月末/季末/年末                        | 自动处理           |

#### 4.5.3 校验执行顺序

1. **字段校验**：必填、类型、格式（前端校验 + 后端二次校验）
2. **业务校验**：外键存在性、权限、周期非未来（后端校验）
3. **重复检测**：表格内重复（前端警告）、与数据库重复（后端按 mergeMode 处理）

### 4.6 错误处理机制

#### 4.6.1 批量处理策略

**逐条提交模式**：

* 后端对 records 数组逐条处理，每条独立成功或失败

* 单条失败不影响其他记录的处理

* 所有记录处理完毕后，汇总返回结果

* 不使用数据库事务包裹整个批量操作（避免一条失败全部回滚）

**处理流程**：

```
for each record in records:
  1. 校验该条记录（字段、外键、权限）
  2. 校验失败 → 标记为 failed，记录错误信息，continue
  3. 查询是否已有记录（formula_id + salesman_id + period_type + period_start）
  4. 无已有记录 → INSERT，标记为 success（action: created）
  5. 有已有记录：
     a. mergeMode = accumulate → UPDATE 累加，标记为 merged（action: accumulated）
     b. mergeMode = replace → UPDATE 覆盖，标记为 merged（action: replaced）
```

#### 4.6.2 前端错误展示

1. **提交前校验失败**：失败行在表格中高亮红色，鼠标悬停显示错误原因，禁止提交
2. **提交后部分失败**：结果对话框展示成功/失败/跳过条数，失败行可展开查看详情
3. **失败行修正**：失败行保留在表格中，用户修正数据后可重新提交

#### 4.6.3 失败恢复机制

| 场景        | 恢复方式                   |
| --------- | ---------------------- |
| 在线表格部分行失败 | 失败行保留在表格中，用户修正后可仅提交失败行 |
| 网络中断      | 前端提示网络错误，已成功录入的数据不受影响  |

## 5. Risks & Roadmap

### 5.1 Phased Rollout

| 阶段 | 内容                                                 | 优先级 |
| -- | -------------------------------------------------- | --- |
| P0 | 数据库唯一约束迁移 + 后端切换 Controller                        | 高   |
| P0 | 重复录入智能合并（前后端）                                      | 高   |
| P1 | 业务员维度录入入口                                          | 中   |
| P1 | 后端 periodStart 校验                                  | 中   |
| P1 | 在线表格批量录入（SalesBatchDrawer + POST /api/sales/batch） | 中   |
| P2 | 复制上期数据快速填充                                         | 低   |

### 5.2 Technical Risks

| 风险                         | 影响               | 缓解措施                                  |
| -------------------------- | ---------------- | ------------------------------------- |
| 唯一约束变更需迁移现有数据              | 若现有数据违反新约束则迁移失败  | 迁移脚本先检测冲突数据，提示人工处理                    |
| Controller 切换可能改变 API 响应格式 | 前端解析失败           | Controller 使用 rowsToCamelCase，确保字段名一致 |
| 批量 API 逐条处理性能              | 200 条数据可能耗时较长    | 设置合理超时（30s），前端展示提交中状态                 |
| 并发批量录入冲突                   | 多人同时批量录入同一配方同月数据 | 依赖数据库唯一约束 + mergeMode 处理，逐条检测避免冲突     |
| 误操作批量覆盖                    | 大量数据被错误覆盖        | 覆盖模式需二次确认，默认累加模式                      |
| 复制上期数据量大                   | 上期记录过多时前端卡顿      | 仅加载当前筛选条件下的配方列表，分批填充                  |
