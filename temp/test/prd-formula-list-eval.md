# 配方列表页面结构改进评估报告

## 1. Executive Summary

### Problem Statement
当前配方列表页采用"三段式"垂直堆叠布局（仪表盘 → 列表 → 动态/助手），在 1920×1080 主流分辨率下，首屏仅能看到仪表盘 + 列表表头，核心列表数据被挤压至第二屏，底部动态区域长期处于可视区域之外。用户反馈"小助手"功能希望增强，但当前结构导致其曝光率不足。

### Proposed Solution
将三段式垂直布局重构为"左右双栏 + 顶部仪表盘"的复合布局：左侧主区域放置配方列表（占 2/3），右侧边栏放置仪表盘 + 小助手（占 1/3），底部动态区域改为可折叠面板或整合至右侧边栏。在保持信息完整性的同时，提升首屏信息密度和核心操作效率。

### Success Criteria
| KPI | 当前基线 | 目标值 | 测量方式 |
|-----|---------|--------|---------|
| 首屏可见列表行数 | 2-3 行 | ≥6 行 | 截图像素测量 |
| 小助手区域曝光率 | <15% | ≥60% | 热力图/埋点 |
| 列表→详情操作路径长度 | 1 次点击 | 保持 1 次 | 不变 |
| 页面加载后用户首次滚动时间 | <3s | 保持 <3s | 性能监控 |
| 仪表盘数据刷新延迟 | 静态（badge 写死） | T+1 自动刷新 | 接口日志 |

### 2.2 各区域详细评估

#### Section 1: 数据仪表盘 (dashboard-grid)

| 维度 | 现状 | 评估 |
|------|------|------|
| 布局 | 4 列 grid，响应式 1024px→2列，768px→1列 | 合理，但 4 列在 1366px 下略显拥挤 |
| 内容 | 活跃配方数、原料库、本月销量、本月销售额 | 信息价值中等，但 badge 数据（+12.5%）为硬编码 |
| 视觉 | 圆角卡片、阴影、hover 上浮动画 | 美观，但每个卡片高度 120px+，占用首屏 25% 空间 |
| 交互 | 纯展示，无点击跳转 | 浪费——用户希望从"本月销量"直达销量明细 |

**关键问题**: 仪表盘数据是"静态快照"而非"动态入口"。当前 `dashboardCards` computed 中 badge 值（如 `+12.5%`）为写死字符串，未对接真实增长计算。用户期望 T+1 刷新，但当前实现无法满足。

#### Section 2: 配方列表 (content-card)

| 维度 | 现状 | 评估 |
|------|------|------|
| 布局 | 单卡片全宽，内部 t-table | 标准列表布局，功能完整 |
| 功能 | 搜索、筛选、批量操作、展开行、分页 | 功能丰富，但展开行内容过多 |
| 列数 | 8 列（选择、名称、状态、原料数、业务员、销量、成本、操作） | 1920px 下合适，1366px 下可能溢出 |
| 展开行 | 配方信息 tags + 版本记录 + 变更详情 | 信息过载，展开后单条高度可达 400px+ |
| 分页 | 自定义分页组件，显示数据量信息 | 合理 |

**关键问题**: 
1. 展开行内容过于厚重（版本记录 + 变更详情），导致列表滚动效率低
2. 成本列显示 `--` 时无引导操作，用户不知如何补录价格
3. 销量列可点击打开抽屉，但视觉提示不足（无下划线/手型光标）

#### Section 3: 近期动态 + 小助手 (activity-section)

| 维度 | 现状 | 评估 |
|------|------|------|
| 布局 | 2:1 grid（左动态右助手），1024px 以下堆叠 | 动态区域占 2/3，但实际价值可能不如助手 |
| 动态内容 | 时间线样式，分页浏览 | 内容来源不明，更新频率未知 |
| 小助手 | 待办列表（警告/提示/成功三级优先级） | 功能有价值，但位置在页面最底部 |
| 交互 | 待办可点击执行操作 | 合理，但用户需要滚动到底部才能发现 |

**关键问题**: 
1. 小助手是用户明确希望"增强功能"的区域，但当前位于页面最底部，曝光率极低
2. 动态区域与小助手的 2:1 比例可能倒置——助手的信息密度和行动价值更高
3. 动态内容缺乏分类筛选（全部/配方变更/原料变动/系统通知）

### 2.3 信息架构问题总结

```
信息层级混乱:
┌────────────────────────────────────────┐
│  P0: 仪表盘 (静态展示)                  │  ← 占据首屏黄金位置，但交互价值低
├────────────────────────────────────────┤
│  P1: 配方列表 (核心功能)                │  ← 被仪表盘挤压，首屏仅见 2-3 行
├────────────────────────────────────────┤
│  P2: 近期动态 (信息补充)                │  ← 内容价值不明确，更新频率未知
│  P1: 小助手 (待办+行动)                 │  ← 用户希望增强，但位置最底部
└────────────────────────────────────────┘
```

**核心矛盾**: 信息优先级与视觉层级不匹配。仪表盘（P0）占用了最宝贵的首屏空间，但用户真正高频操作的是列表（P1）和待办（P1）。

## 3. 改进方案设计

### 3.1 方案 A: 左右双栏布局（推荐）

```
┌─────────────────────────────────────────────────────────────┐
│  [顶部导航栏]                                                │
├──────────────────────────────┬──────────────────────────────┤
│                              │  ┌────────────────────────┐  │
│  配方管理中心                 │  │ 📊 数据仪表盘           │  │
│  [搜索______] [创建] [筛选]   │  │ 活跃配方: 128款 ↑12.5% │  │
│                              │  │ 原料库: 86种 持平       │  │
│  ┌────────────────────────┐  │  │ 本月销量: 2,450件 ↑8%  │  │
│  │ 配方列表                │  │  │ 本月销售额: ¥12.5万    │  │
│  │ ┌────┬────┬────┬────┐  │  │  └────────────────────────┘  │
│  │ │名称│状态│原料│操作│  │  │  ┌────────────────────────┐  │
│  │ └────┴────┴────┴────┘  │  │  │ 🤖 配方师小助手         │  │
│  │                        │  │  │ ⚠ 3个配方待审核 →      │  │
│  │ [分页]                  │  │  │ ⚠ 2个原料缺单价 →      │  │
│  └────────────────────────┘  │  │ 📋 近期动态（折叠）▼    │  │
│                              │  │ • 配方A已更新           │  │
│                              │  └────────────────────────┘  │
│         主区域 (65%)          │        侧边栏 (35%)          │
├──────────────────────────────┴──────────────────────────────┤
│  [底部可选: 快捷操作栏/全局提示]                              │
└─────────────────────────────────────────────────────────────┘
```

**改动点**:
1. 仪表盘从顶部移至右侧边栏顶部，缩小为紧凑卡片组
2. 配方列表占据左侧主区域 65%，首屏可见 8-10 行
3. 小助手移至右侧边栏中部，曝光率提升至 60%+
4. 近期动态改为可折叠面板，默认收起，节省空间
5. 响应式: 1024px 以下恢复为垂直堆叠

**优点**: 首屏信息密度高，核心功能和小助手同时可见，符合用户"两者兼顾"的使用场景
**缺点**: 右侧边栏需要滚动，仪表盘和小助手可能无法同时完全可见

**改动点**:
1. 仪表盘从 4 卡片 grid 压缩为单行指标栏，支持收起/展开
2. 列表区域全宽，首屏可见 6-8 行
3. 工具栏新增"小助手"快捷入口（带未读 badge），点击展开抽屉
4. 底部保留动态和助手，但助手可通过工具栏快捷入口直达

**优点**: 改动量小，列表区域最大化，小助手有多个入口
**缺点**: 仪表盘收起后数据可见性降低，底部助手仍存在曝光问题

## 4. 技术规格

### 4.1 推荐方案: 方案 A（左右双栏）

**布局实现**:
```scss
.formula-list {
  display: grid;
  grid-template-columns: 1fr; // 移动端默认单列
  gap: 24px;

  @media (min-width: 1280px) {
    grid-template-columns: 2fr 1fr; // 主区域 : 侧边栏 = 2:1
  }
}

.main-area {
  // 列表区域
  min-width: 0; // 防止 grid item 溢出
}

.sidebar {
  // 侧边栏: 仪表盘 + 小助手 + 动态
  display: flex;
  flex-direction: column;
  gap: 24px;

  @media (min-width: 1280px) {
    position: sticky;
    top: 24px;
    align-self: start;
    max-height: calc(100vh - 48px);
    overflow-y: auto;
  }
}
```

**组件拆分**:
```
FormulaList.vue
├── MainArea/
│   ├── FormulaToolbar.vue      // 搜索、创建、筛选
│   └── FormulaTable.vue        // t-table + 分页
└── Sidebar/
    ├── DashboardMini.vue       // 紧凑仪表盘（4指标→2×2 grid）
    ├── AssistantPanel.vue      // 小助手（原 activity-card--assistant）
    └── ActivityTimeline.vue    // 近期动态（可折叠）
```

### 4.2 数据流调整

| 数据 | 当前刷新频率 | 建议刷新频率 | 实现方式 |
|------|------------|------------|---------|
| 配方列表 | 页面加载 + 分页/搜索 | 不变 | Store fetch |
| 仪表盘销量/销售额 | 静态（写死 badge） | 每日凌晨 3:00 | Cron job + API 缓存 |
| 小助手待办 | 页面加载 | 每 5 分钟轮询 | setInterval + API |
| 近期动态 | 页面加载 | 实时 WebSocket 推送 | 或轮询 30s |

### 4.3 响应式断点

| 断点 | 布局 | 侧边栏处理 |
|------|------|-----------|
| ≥1280px | 2:1 双栏 | 右侧 sticky 侧边栏 |
| 1024-1279px | 单列，侧边栏在列表下方 | 水平排列仪表盘+助手 |
| 768-1023px | 单列 | 仪表盘折叠为指标行 |
| <768px | 单列 | 仪表盘收起，助手变为浮动按钮 |

## 5. 风险评估与路线图

### 5.1 风险矩阵

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|---------|
| 侧边栏内容过多导致滚动冲突 | 中 | 中 | 侧边栏独立滚动 + 最大高度限制 |
| 小屏设备（<1280px）体验下降 | 高 | 中 | 强化 1024px 以下单列布局优化 |
| 用户不习惯新布局 | 中 | 高 | A/B 测试 + 可选切换旧布局（30天过渡期） |
| 仪表盘 API 实时化开发成本 | 低 | 中 | 第一阶段保持静态，第二阶段接入定时任务 |
| 展开行内容迁移至详情页的成本 | 中 | 中 | 分阶段迁移，先移版本记录，保留配方信息 tags |

### 5.2 分阶段实施路线

**Phase 1: 布局重构（2 周）**
- 实现左右双栏布局（方案 A）
- 仪表盘组件提取为 DashboardMini.vue
- 小助手提取为 AssistantPanel.vue
- 保持所有现有功能不变
- 添加布局切换开关（旧/新），默认新布局

**Phase 2: 功能增强（1 周）**
- 仪表盘 badge 数据接入真实增长计算（T+1）
- 小助手添加轮询刷新（5 分钟）
- 近期动态添加分类筛选标签
- 列表展开行内容精简（版本记录移至详情页）

**Phase 3: 体验优化（1 周）**
- 响应式断点微调（基于用户设备分布数据）
- 侧边栏滚动行为优化（独立滚动条、滚动位置记忆）
- 移除旧布局切换开关
- 性能监控埋点

## 6. 非目标（Non-Goals）

以下功能**不在本次改进范围内**：
- 配方列表的虚拟滚动（当前数据量 <1000，无需优化）
- 仪表盘数据的实时推送（WebSocket，成本过高）
- 小助手的 AI 智能推荐（需独立项目规划）
- 列表列的自定义拖拽排序（TDesign table 限制）
- 深色模式适配（全局功能，非列表页独占）

## 7. 结论与建议

### 核心结论
当前三段式垂直布局在信息层级上存在"头重脚轻"的问题：仪表盘（静态展示）占据首屏黄金位置，而用户高频操作的列表和待办助手被挤压至下方。建议采用**方案 A（左右双栏布局）**进行重构。

### 关键改进点
1. **首屏效率**: 列表从 2-3 行提升至 8-10 行可见
2. **助手曝光**: 小助手从 <15% 曝光率提升至 60%+
3. **数据时效**: 仪表盘从静态写死提升至 T+1 自动刷新
4. **信息密度**: 右侧边栏 sticky 定位，滚动时始终可见
